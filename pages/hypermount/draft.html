一場美麗的錯誤

草草趕工出來的程式語言，反而造就美麗的錯誤。

趕工出來的程式語言

本單元（指超鏈結界）是站在「物件導向程式設計」角度整理出來的語言用法。

隨著 HTML5 需求的加溫，大家在應用 JavaScript 的過程，開始感受到它 literal notation 寫法的方便，以及函式是一級物件的魅力。因此漸漸地本單元所介紹的語法，到後來可能會被嫌棄，認為既然用 JavaScript 就該展現自己特有的卓越語法特性，不該用 Java 的習慣來使用它；事實上，連 Java 都開始引進某些一旦上手後就難以割愛的 JavaScript 語言特性。

有趣的是，在 JavaScript 對軟體工程來講，因為原始碼在團隊場合不容易管理，所以很難被接受說將它導入到大型開發案的時代，這些語法是每個軟體工程師看了都詬病不已的爛寫法。不得不硬著頭皮導入 JavaScript 寫了一陣子程式後，反而迷上了它的寫法，甚至悄悄地在改變傳統物件導向程式語言的習慣。

期待快點看到時機成熟，在大家都接受與習慣 JavaScript 特有的寫法時，本單元能夠改寫 JavaScript 語法，見識這是多麼有趣的程式語言…雖然那也是災難的開始，團隊的原始碼管理其實至關重要，寫得的時候很有趣，維護的時候卻很痛苦，對產業來說是相當嚴峻的考驗。

誕生

網際網路剛迅速崛起的時代，最惱人的使用者體驗，就是提交資料給某個全球資訊網後，等了好幾秒終於看到下一頁時，竟然是回報說送出的資料錯誤。為了提高瀏覽者與全球資訊網的互動效率，Netscape Communications 研擬 Mocha 計畫，希望瀏覽器本身就能跑一些簡單的程式，來分擔伺服器的工作，與其浪費連線時間獲得錯誤回報，不如事先確認好再連線遞交。

Netscape Communications 委派 Brendan Eich 開發一款新的程式語言，它能夠將原始碼直接寫在網頁裡面，而且沒寫過程式的人也能使用。

沒多久 Netscape Communications 對 Applet 的表現感到迷戀，與 Sun Microsystems 合作，除了讓自己的瀏覽器支援 Applet 外，還要讓開發中的程式語言長得像 Java。

Brendan Eich 參考 Self 程式語言的原型機制，以及 Scheme 程式語言的一級函數，只花約十天的時間，完成了一款弱型別的直譯式程式語言順利交差。

1995 年 9 月先以 LiveScript 的名稱，搭載於 Netscape Navigator 2.0 測試版，12 月正式版時改名為 JavaScript。（至於 Mocha 是程式庫，即後來的 SpiderMonkey 引擎。）

風靡

基本上這是一款可以操作些瀏覽器軟體本身的指令語言，簡簡單單一兩行程式，無須專業程式設計師背景，就能動態變化瀏覽器內容，顯著可見的效果讓 JavaScript 大受好評！大家都搶著在網頁寫個 JavaScript 來露一下身手。

於是 1996 年 8 月 Microsoft 趕緊也抄了一個 JScript 來應戰，推出 Internet Explorer 3.0。

但 Netscape Communications 並不介意對手的抄襲，反而基於更有效推廣這款程式語言，來帶動全球資訊網的進化，同年 11 月將 JavaScript 提交給歐洲電腦公會，制定統一規範，於是產生了 ECMAScript。

興衰

不過挾帶 Windows 作業系統市占率破九成優勢，覺得自己就是標準的 Microsoft 並沒有這麼先進文明的觀念，往後照樣以自己高興就好的態度破壞規則，甚至故意用不相容的做法來綁定 Windows，企圖邊緣化無法與之公平競爭的對手，完全不鳥 ECMAScirpt 和日後的 W3C HTML。

原本與 Netscape Navigator 瀏覽器競爭時，進步速度很快的 Internet Explorer 瀏覽器，在 4.0 版幹掉對手，進入 5.0 版發現沒人需要競爭後，就不再進步了！2003 年超過九成的民眾，都是用 Internet Explorer 瀏覽器，但像最基本的分頁瀏覽，都過幾年了，還懶得設計出來給你用，於是越來越多內行的人選擇 Mozilla Firefox 來獲得更高的瀏覽體驗，並形成一股還在用 IE 表示他電腦很外行的觀感。

加上 IE9 推出時沒有 Windows XP 版：「其實 IE7 推出時，也沒有 Windows 2000 版。」大家終於意識到該換其他瀏覽器才能確保自身權益，Internet Explorer 的市佔率終於被 Mozilla Firefox 明顯侵蝕掉。

正當 Internet Explorer 市佔率節節下滑之際，隨即又被不為惡、但很偽善的 Google 官方瀏覽器，以簡單清爽的介面，及世界第一的 JavaScript 飆速，確實提升 Web 2.0 體驗，更快吃掉 IE 市佔率。

反 IE 成為顯學的時代，才在 IE9 宣稱自己遵守 HTML5 規範，企圖挽回聲譽。而 Windows 8 的 IE10 更是注重在網頁整體的載入效率[1]，就瀏覽全球資訊網來講，其實比只顧飆 JavaScript 速度的 Google Chrome 更快處理好整張網頁。但這時已進入平板電腦行動上網的時代，Safari 和 Chrome 這以 WebKit 核心的瀏覽器正在醞釀成為網際網路主流的趨勢，已經沒人想去關注 IE10 是否真的超越了誰。

主流

經過幾番風雨，JavaScript 現在是進階存取 HTML5 文件結構的根本技術，不再只是特效集玩玩而已，它成為擔負起 HTML5 的…（無限期草稿中）

《網頁平台的沿革》Internet

interconnection network。Internet 是一種硬體通信設備系統，將全球電腦連接起來，形成一種通信網路；台灣通稱「網際網路」。

「通信」與「通訊」

這裡「通信」與「通訊」是有差異的[1]，通信傳遞的是「信息」，通訊傳遞的是「訊息」。

Internet 主要傳遞的是機器之間交換資料的信號，而不是提供內容的訊號。

當然，Internet 傳遞信號時，依然包含了內容，但如何表現內容並不是 Internet 的主要工作。

服務

藉由 Internet 通信的機制，提供更明確內容之傳遞服務的，最有名者像是 HTTP 與 FTP，更進一步以表現內容為服務的，則像「全球資訊網」和「即時通」。

全球資訊網

全球資訊網，英文 World Wide Web，簡稱 WWW，是最熱門的「網際網路」服務，甚至時常被誤以為「網際網路」的代名詞！ 全球資訊網使用 HTTP 與 URI 做為網路的系統架構，然後以 HTML 做為資料內容的結構。這種資料即「網頁」，以圖文並茂的資料內容發布於網際網路，由於以文件的形式表現內容，易懂易用，廣為全球民眾接受，成為現代電腦最重要的一環。

[1] Internet 傳遞的信號像是 Singal，WWW 傳遞的訊號像是 Information。

《網頁平台的沿革》Browser

網際網路的各項服務，都有對應的工具程式，其中專門用來瀏覽全球資訊網的，就叫「瀏覽器」，英文 Browser。

網頁瀏覽器

由於全球資訊網的目的是呈現網頁以供瀏覽，因此瀏覽器也可以說是專門用來呈現網頁的應用軟體。

網頁

網頁的本體，使用 HTML 語法來製作，並且可以使用 CSS 來重新定義 HTML 的表現樣式，還可以透過 JavaScript 以程式設計的手法來操作網頁。

網頁平台

換個角度想，網頁瀏覽器可視為用來執行 JavaScript 的程式平台，以 HTML 和 CSS 做為這種程式平台的操作介面。

在不同裝置安裝網頁瀏覽器是相當容易的，因此使用瀏覽器的平台來發展 HTML 為基礎的應用程式，不失為解決裝置差異問題的方案。

《網頁平台的沿革》HTML

Hyper Text Markup Language，超文件標記語言。

最早以前，要在網際網路發表學術文章，都會寫成 HTML 格式的文件，然後透過全球資訊網來散佈。[1]

超文件

超文件是提供「超鏈結」的文件，可讓使用者在網際網路中連結其它全球資訊網的內容。

搭配各式顯示文字、製作表格、載入圖片的標記，可以製作圖文並茂的文件。

往後持續加強輸入表單、排版容器、載入影音的標記，現在超文件除了適合做為平面設計的宣傳平台，也適合做為多媒體設計的傳播頻道。

[1] 然而隨著網路時代蓬勃的發展，延伸出許多種語法，HTML 已經成為一種概念性名詞，泛指整體的網頁技術。
[*] 超文件真正通俗的稱呼是「網頁」，英文：Webpage。

《網頁平台的沿革》JavaScript

網頁載入以後就無法再變動內容，想要變動內容除非重新載入，所以是靜態的。

為了能夠變化載入後的網頁內容，而推出類似程式設計寫法的 JavaScript 語言，並開啟動態網頁的時代。

DOM

Document Object Model，文件模型。

在還沒有 DOM 時，使用 JavaScript 寫出來的功能，往往無法在不同瀏覽器執行[1]，只能選邊站，可說實用性不高。直到負責維護 HTML 標準規範的 W3C 制定 DOM，才總算解決 JavaScript 寫法的實用性[2]。

此外，DOM 還規範出以樹狀節點來處理網頁裡所有元素的做法，讓 JavaScript 可以動態增加、刪除、更改網頁的標記，更是讓 JavaScript 對網頁內容的控制能力全盤提升。

DOM 的出現，讓 JavaScript 可以完全控制網頁內容的 HTML 與 CSS，奠定網頁程式平台的里程碑。

[1] 尤其當時 Microsoft 一副 Windows 就是一種標準規格的姿態，旗下 Internet Explorer 瀏覽器狂推一堆只有自家環境能用的功能，無視標準規格追求的相容性議題，可說是禍害。
[2] 改以文件內容為動態改變的對象，比從瀏覽器這個對象來改變內容，要合理多了！因此，雖然 DOM 與 JavaScript 語法無關，但能有這麼一個讓我們將設計對象聚焦在文件內容本身的 API，對 JavaScript 標準寫法有很大的幫助！這也是一種實用性的表現～

《網頁平台的沿革》CSS

Cascading Style Sheets，階層樣式表。

用來重新定義 HTML 標籤的表現方式。

樣板

CSS 可以寫在另一份文件，然後讓不同網頁載入的特性，適合做為制定樣板的工具，而大量被使用。

從延伸到必備

到了 HTML5，幾乎取消 HTML 標記原先具備的屬性，改由 CSS 來呈現這些效果，使得原本像是延伸出來、可用可不用的 CSS 變成 HTML 必備的項目。[1]

[1] 然而這卻增加初學 HTML 的複雜性，因此建議初學者先從 HTML 4.01 著手，認識標記與屬性的功效，然後慢慢導入 CSS。深刻體會到 CSS 的實用性與方便性的確是不可或缺的，再轉換到 HTML5。瀏覽器可以正常顯示 HTML 4.01 和 HTML5 混雜在一起的寫法，所以這點不用擔心。

《網頁平台的沿革》Java Applet

字尾 -let 有「小的」意思，因此 Applet 意思是迷你的 Application（應用程式）。

Java 藉由跨平台的能力，將 Applet 嵌入網頁來執行，為靜態網頁的時代，提供諸如播放連續圖片、檔案操作等豐富的應用。

跨平台

Java Applet 雖然有跨平台的優勢，但其實不是每個作業系統都能順利在瀏覽器執行 JRE，要解決瀏覽器無法執行 JRE 的問題往往一種折騰人的難題，甚至無法解決。

因此 Java Applet 並不是很好的跨平台解決方案，使用 Applet 做為跨平台的操作介面，存在相當大無法執行的風險。

實用性

Java Applet 為保護網路安全性，可以使用的程式開發功能沒有 Java Application 來得靈活，或者應該說是限制一堆而綁手綁腳才對。

後來出現了更靈活且方便使用的 Macromedia Flash，相形之下難搞的 Java Applet 顯得不具實用性，甚至是拖累開發進展的選擇。

由於 Java Applet 必須執行於網頁，因此應用上就是為網頁提供更多功能。但限制過多的話，開發時效不高，反而用更適合為網頁打造功能的 CGI 或 ASP 要比 Java Applet 來得有效而實用。

相容性

不同版本 JRE 有相容性的問題，新版 JDK 做好的 Java Applet 無法在舊版 JRE 執行情有可原，但舊版 JDK 做好的 Java Applet 可能無法在新版的 JRE 執行就很嚴重了。

另外，瀏覽器對 Java 的實作方式，也會產生相容性問題，舊版瀏覽器原本可以正常執行的 Java Applet，在新版的瀏覽器可能無法正常執行。

建議

Java Applet 是種不確定因素過高的解決方案，在 HTML5 的時代已經沒有使用的必要。

要讓 Java Applet 確定能正常執行，只能設計些簡單的東西，但簡單的東西 HTML5 就能做得很好。新穎的功能又存在版本相容性的風險、以及開發效益低落的問題，使用其它解決方案反而是更有保障的選擇。

《網頁平台的沿革》Flash

Flash 是能夠在網頁播放流暢動畫的軟體，而且還能設計出與觀看者互動的程式功能，讓網頁提供更多樣化的操作方式，使網際網路有了更充實的應用價值。

Macromedia Flash 與 Flash Player

在作業系統安裝 Flash Player 這款程式，瀏覽器就能執行 Macromedia Flash[1] 製作出來的影片。

Flash Player 是開放在網際網路免費下載的，但 Macromedia Flash 則是市售商品。也就是說，雖然每個人都可以播放 Flash 檔案，但只有購買 Macromedia Flash 的人才能製作 Flash 檔案。

RIA 與 HTML5

Flash 能夠設計出與使用者互動的動畫，徹底改變靜態網頁呆板的操作，迅速成為網頁設計最喜歡應用的技術，網頁也因此進入 RIA 的時代。

RIA 是 Rich Internet Application 的縮寫，意思是「為網際網路提供更多樣性的應用模式」。

雖然 Flash 是設計 RIA 的主流做法，但必須購買 Macromedia Flash 才能設計 RIA，因此在網際網路進入 RIA 的時代，能夠實現 RIA 的 HTML5 將滿足網頁設計的需求，並成為推動網際網路進入下一個蓬勃發展時代的新興技術。

迥異的設計模式

對程式設計師來說，Macromedia Flash 的 ActionScript 是種做法特別迥異的程式設計，往往摸不著頭緒，搞不清楚 Macromedia Flash 的程式應該怎麼寫？

這是因為 Macromedia Flash 的使用概念，主要是「一個影格」保存「一張圖畫」，做好的東西執行時，其實是在跑連續個影格，就像製作「卡通」一樣。ActionScript 只是進階控制這些卡通圖畫的指令，例如要跳躍到哪一個影格？圖畫上哪個元件要反應使用者的互動？

因此，只要知道 ActionScript 其實是用來控制 Macromedia Flash「整套軟體」的「指令」，而不是拿來設計「應用程式」的「程式語言」，於是把重點放在這套軟體提供了那些製作卡通的操作介面，一切就會好多了！

[1] Macromedia 已被 Adobe 收購，因此現已改稱 Adobe Flash。

《網頁平台的沿革》Dynamic HTML

引進 Document Object Model 的概念，使腳本語言如 JavaScript 能夠從客戶端存取 HTML 元素與 CSS 樣式的狀態，以動態存取網頁內部資料的技術。通常簡稱 DHTML。

客戶端與伺服端

Dynamic HTML 的優點在於強化客戶端的程式控制能力，能將伺服端的工作分擔給客戶端執行。

《網頁平台的沿革》Ajax

Asynchronous JavaScript And XML 的簡寫，意思是「非同步的 JavaScript 與 XML」，能在不重新整理頁面的情況下存取伺服端的資料，使網頁具備即時顯示網站資料的特性。

XMLHttpRequest 與 XML

Ajax 主要運用 XMLHttpRequest 為核心技術，並且使用 XML 做為資料交換的格式。

破壞瀏覽器的操作慣例

Ajax 以類似背景執行的程式方式即時更動頁面的內容，這對全球資訊網以上一頁和下一頁來紀錄前後內容的慣例有很大出入，因此設計 Ajax 需要注意瀏覽網頁時，被按上一頁後要如何維持資料，避免出現與期待不符的操作慣例造成使用者困擾。

《網頁平台的沿革》Web 2.0

第二世代網站應用模式。

網際網路的發展，從單方面提供資訊讓人閱讀，逐漸演變為互動的交易平台。一開始只是注意到網路走向這個趨勢，並無法明確指出這會是怎樣的情況，因此提出 Web 2.0 的概念，希望促進網站技術的整合，早日看到第二世代的網站應用模式。

目前這個趨勢已經發展成熟，清楚看到 Web 2.0 其實就是網站 (Web) 應該是一種提供服務 (Service) 的平台。

雲端

設計圖中，網路的部分會畫作雲狀來表示，因此透過網路來運算的程式架構，被稱為「雲端」。

消極的雲端解釋

表面上來看，所謂雲端是把資料儲存在網路伺服器，這樣即使轉換任何裝置、以及在任何地方使用電腦，都能使用個人化的資料，繼續完成想要的工作。

積極的雲端定義

從細節來探討，雲端是串連多台大型主機來執行商業級運算，再透過網路把運算好的結果傳給使用者。讓即使沒有工作站等級的電腦使用者，也能享受高檔規格的應用環境。

然而網路有延遲的瓶頸，尚有待突破性技術來解決，目前還是鮮少透過雲端執行大量運算的服務。現在雲端服務都以簡單的動做為主，甚至沒在串連電腦來執行運算的網站，也自稱雲端服務，導致使用者對這個名詞的混淆。

理想的雲端應用程式

一個理想的雲端應用程式，就是應用程式本身像是個空殼子，本身沒什麼運算的功能，只提供操作介面，讓使用者操作伺服端的運算服務。

目前平板電腦流行的 App，很多只是把 WWW 網站的功能，重新包裝成應用程式的樣子來存取，這就是一個很好的雲端應用程式實作範例！

ECMAScript 5 其實是過度時期的版本

ECMAScript 5 新增了 strict mode 的機制，可以讓解譯器警告一些積習已久、即將被淘汰的程式寫法。只要在程式碼最前端標上 'strict mode' 這個字串即可！這樣執行 JavaScript 時，如果程式碼的寫法觸犯長久以來的陋習，會直接當作語法錯誤無法執行。

甚至可以這麼說，ECMAScript 5 增加的功能，像是原型物件的映射、標準支援 JSON 的格式、一級函式物件的銳化…是為了讓我們解決 strict mode 時的語法錯誤，才放上來的。嗯，這說法真是太穿鑿附會了～！但確實還有很多語法沒在 ECMAScript 5 和 ECMAScript 5.1 用上，目前看到的，都是試圖指導我們原型物件有那些好處的功能。

這告訴我們，真正值得期待的，是目前代號 Harmony 的 ECMAScript 6！

然而，我期待 ECMAScript 6 不怎麼相容過去的版本，寧可改寫程式碼符合 ECMAScript 6 的規範，也比持續 ECMAScript 累積下來的陋習所造成弊端來得有效率；越不相容，越能一次解決 JavaScript 的問題。

修改 JavaScript 程式碼通過 ECMAScript 6 警示的語法錯誤並不難，反正你發出幾條錯誤，我就逐行修改過來，程式設計師本來就是在幹這些工作，感覺就跟消除一行又一行的俄羅斯方塊一樣在玩益智遊戲而已。但改完之後，卻能讓我的網頁應用程式日後變得更容易維護。

當然，這是不可能的事，如果這樣就不會是 ECMAScript 5 到 ECMAScript 6 的規範了，通常這叫另一個規範。

應用網頁的時代

為了讓網頁順利成為網路應用程式的操作介面，制定了 HTML5。

但 HTML5 大量取消掉 HTML 的功能，改用 CSS 來取代，這對新一輩正想學習 HTML 的人來講，變成不只要學 HTML 有哪些標記可用，還要學 CSS 才能變換花樣，而這兩樣語法相當迥異，無疑是種負擔。

加上 HTML5 的多媒體功能，像是 CANVAS、AUDIO、VIDEO，光靠 HTML 和 CSS 是無法充分發揮效用的，必須使用 JavaScript 才能控制這些功能，這又讓所謂的 HTML5 更加困難了。

網頁設計師何去何從？

當然，HTML5 的推出只是為了讓網頁可以變成應用程式，但不代表網頁真的只能這樣寫，你要繼續用 HTML 4 或者更早的規範去寫網頁，照樣可以動作；瀏覽器會相容各種規範的 HTML。

只能說，如果你要當一名網頁設計師，只能寫 HTML 4 的網頁，依然可以好好當個平面設計師，學會 HTML5 是為了能夠扮演多媒體設計師的角色。

就這點來看，無論 HTML5 或 HTML 4，需求上依然是相容的，網頁適合做為平面宣傳的功能，不會因為走向多媒體的展現而被取代。但確實會有此消彼長的情形，如果多媒體的互動可以產生更多宣傳效果，平面設計式的網頁就會沒落。

然而無論是平面設計或者多媒體設計，都是不同的專業領域，就算未來網頁走向多媒體互動為主流模式的時代，只要你平面設計的專業有口碑，你的網頁依然有很高的商業價值。所以你應該正視的問題並不是 HTML 本身，而是你真正的專業領域是否夠權威。

配合搜尋環境最佳化網頁內容的語意

技術上 HTML5 等於 HTML + CSS + JavaScript，但應用上有更大的意義：語意。

為了讓各大搜尋引擎能夠更明確解析你網頁的資料，提高資料搜尋的實用性，使用 HTML5 明確劃分自己的網頁內容有哪些成分，有助於未來網際網路的搜尋效益。

事實上這才是 HTML5 的重點！HTML5 很努力定義網頁有哪些成分，除了新增 SECTION（第幾單元）、ARTICLE（第幾章節）、NAV（導覽頁籤）、ASIDE（邊角專欄）、HEADER（標題列）、FOOTER（頁腳）…等標籤，甚至原本只是用來呈現水平線的 HR 標籤，HTML5 都進一步定義為段落的變化，規範網頁成分的用心非常強烈。

底下是配合 HTML5 語意所寫出來的網頁文件範例：

HTML + CSS + JavaScript, Model–view–controller.

HTML + CSS + JavaScript 這三種不同實作方式的整合，本身就隱含 MVC 模式的特性；這個議題是值得投注的。

甚至如果這樣整合的實作方案，哪天被驗證說是更巧妙的 MVC 架構，不失為業界莫大改革的助益。（但恐怕很難，我只能確定 HTML + CSS + JS 可以是一種 MVC 模式的表現方式，但這樣一種方式實不實際？不是經驗粗淺的我能夠奠定的。）

HTML5 無法興起的原因

以智慧型手機為例，雖然 HTML5 有語法的標準化，可以達到跨硬體裝置、跨軟體平台的優勢，但各家廠商還是喜歡使用原生程式寫 App。

因為 HTML5 很難優化，而廠商怕的是程式跑不順，不是程式寫得快，所以寧願在不同裝置、不同平台，以不同的原生程式開發 App，也比為 HTML5 寫 App 時，遇到效能瓶頸很難解決要好。

有多難？寫 HTML5 平台的應用程式，在 A 裝置好不容易偷吃步解決效能瓶頸時，通常到了 B 裝置並不適用，必須重新找偷吃步的方法。沒辦法，誰叫你偷吃步解決效能瓶頸，這樣就不夠 HTML5 了！但遺憾的是，不偷吃步解決不了效能瓶頸的問題～

而各裝置提供的原生程式，總是能基於自己裝置的硬體特性，提供豐富的 API 幫忙解決問題，因此還不如 A 裝置就用 A 的原生程式、B 裝置就用 B 的原生程式要來得省事。

vi：一款執行於指令模式的文字編輯器

網管人員往往需要在指令模式下操控系統，這時如果要檢視或修改設定檔，就需要能執行於指令模式的文字編輯器。

vi 就是一款指令模式下的文字編輯器，你可以透過它載入文字檔來檢視內容，並且對內容進行修改。

vi 最大的優點就是很多伺服主機的作業系統底下都有內建這款文字編輯器，算是一款通用的工具，可做為網管人員必備的標準事項。

缺點就是沒學過很難上手，例如要複製一個單字是以輸入 yw 的方式來進行，與平常的「Shift + 方向鍵」然後按「Ctrl + C」做法不同。

本篇內容將告訴你，如何使用 vi 載入設定檔，然後檢視其內容、修改文字、最後儲存起來。

vi 快速攻略

本章節提供快速上手的功能指引。

執行 vi 文字編輯器

由於每個 Linux 發行版都會內建 vi 文字編輯器，因此你只要在指令模式的輸入欄位鍵入 vi 然後按 Enter，就可以執行 vi 文字編輯器。

認識 vi 文字編輯器的畫面

執行 vi 之後，你會看到一個純粹由文字與字元符號所構成的畫面。

這畫面看起來雖然很簡陋，但其實分為三個控制區域可以切換，分別執行不一樣的任務，簡介如下：

1. normal mode - 標準模式；可檢視內容，且透過按鍵的組合方式，可以執行 vi 在軟體部份的功能。
2. insert mode - 插入模式；可編修文字。
3. command-line mode - 命令列模式；用來開啟、儲存檔案，並可執行搜尋、替換的工作。

執行 vi 程式後，控制區域會先位於 normal mode。這時你可以按 i 鍵進入 insert mode，或者按 : 鍵進入 command-line mode。

在 insert mode 要返回 normal mode，請按 Esc。如果是 command-line mode，只要執行命令就會自己回到 normal mode。

開啟舊檔

請進入 command-line mode，首先鍵入 n，然後隔個空白後，再輸入檔案名稱，即可開啟該檔案。

檢視內容

在 normal mode 即可檢視檔案內容。

你可以使用方向鍵移動游標。

編修內容

在 normal mode 檢視內容的過程中，你可以隨時切換到 insert mode，這時游標會保持在你檢視內容的位置，讓你直接從該處修改內容。

儲存檔案

當你編修好內容想要儲存起來，請切換到 command-line mode，鍵入 w，然後按 Enter，即可將檔案儲存起來。

結束 vi 文字編輯器

當你修改過檔案內容，並且已經儲存的話，那只要在 command-line mode 鍵入 q，然後按 Enter，就可以結束 vi 文字編輯器。

雖然 vi 的功能完整，可做為開發用的文字編輯器，但操作方式過於複雜，就算全部功能都學起來，也不見得會想用來撰寫程式碼或系統設定。

Mozilla Firefox 5 的 localStorage 問題

【聲明】新版本已修訂這個問題，本文僅保留為追蹤研究用，藉以了解瀏覽器的運作。

用 Firefox 5 直接從磁碟開啟內含 localStorage 來儲存資料的網頁，會發現重新載入網頁後，資料就消失了，並未保存起來。

原因

直接從磁碟開啟網頁，屬於使用 file 通訊協定的行為，這不具備 domain 的特性，因此 Mozilla Firefox 5 認為 localStorage 不應該生效。

解決辦法

1. 本地端的網頁無法使用 localStorage 保存資料，必須放在伺服端。例如網頁改放在 http://127.0.0.1 來鏈結就能保存資料，直接在桌面或資料夾開啟網頁無法保存資料。

這樣無法保存資料：file:///D:/mywebpage.html
這樣可以保存資料：http://127.0.0.1/mywebpage.html
但這樣就不叫本地端應用程式了！網頁只能放在架設的伺服器裡面，而不是隨便放都能點來用。

2. 改用 Cookie 保存本地端的資料，但這次換 Google Chrome 不允許儲存本地端的 Cookie　－－

3. 設計本地端應用網頁的話，建議使用者改用非 Firefox 的其它瀏覽器。因為這問題已經不是第一次發生，可見 Mozilla 堅持這樣才是最正確的做法，所以只好改用其它更友善、懂得從善如流的瀏覽器。事實上只有 Firefox 的 localStorage 無法保存本地端網頁的資料。

探討

可見即使進入 HTML5 時代，網頁應用程式的設計，依然無法達到完全的相容性，很有可能設計上遇到瓶頸，最終結果依然是選邊站，變成只能用特定瀏覽器才能執行你的設計。

從 OpenSSL 的 Heartbleed 漏洞，看開放原始碼的安全性問題。

2014 年 4 月 8 日，OpenSSL 正式通報名為 Heartbleed 的資安漏洞，這個臭蟲一旦被攻擊會造成記憶體資料的洩漏，讓駭客取得長達 64K 的資訊，這足以獲得任何他想要的加密內容。原本加密傳輸過程應該保護的資料，像是應該僅限主從兩端傳遞的網頁內容、帳號密碼、甚至連金鑰都能取得，信用卡裡面機密也難逃一劫。

含有該漏洞的 OpenSSL 1.0.1 是在 2012 年三月發布，已經長達兩年之久，加上不是惡意攻擊，比較像大剌剌取得金鑰的使用者來獲取資訊，因此無從判斷行動以及追蹤來源，被稱為史上最嚴重的漏洞！

幸好一個星期多來，各界都搶在最短時間內應變，像是緊急關閉伺服器，直到確認哪些細節有用到 OpenSSL 加已更新後才開機，還未傳出毀滅性資安災難，只有零星的攻擊事件。

我一直認為，開放原始碼受全世界使用者的檢視，只要有問題就會被立刻挑出來，因此是最安全的！但今天看到，連 OpenSSL 這全球超過 60% 使用率、幾乎是架 Linux 和 Apache 網站就一定採用的網路加密技術，都能潛藏兩年之久的系統漏洞沒被察覺：「換句話說全世界使用者其實沒人會去檢查原始碼，一切還是靠專案管理團隊效勞。」所以開放原始碼真的安全嗎？甚至開放出來誰都能改的關係，專案團隊管控不嚴密的話，開放原始碼說不定反而是不安全的選擇。例如美國國家安全局為了監視全球網路，故意派人假裝修正原始碼，其實偷偷故意放一些後門進去的話，其實一點也不安全～

Heartbleed 臭蟲是德國研究生 Robin Seggelmann 在遞交改善過的 OpenSSL 程式碼時，疏忽檢查特定資料的長度，留下可能導致造成記憶體溢位的問題。審查者 Stephen Henson 也沒有發現到這樣的疏失，就這樣包含在 OpenSSL 1.0.1 函式庫裡長達兩年之久，後來才由 Neel Mehta 發現。

而 OpenSSL 這項專案的組織，每年只能獲得 2000 美元的捐助金可以運用，人力物力其實是不足的。

在這平板電腦即將取代桌上型電腦與行動電話的網際網路通訊時代，盼望各國政府能更重視網路通訊的資安問題，提供這些安全性攸關的開放原始碼組織更有力的資助，做為邁入網際網路時代的重要基礎建設～

針對 HeartBleed 成立了 CII 計畫

Heartbleed 幾乎動搖了 IT 產業，Amazon、Cisco、Dell、Facebook、富士通、Google、IBM、Intel、Microsft、NetApp、Qualcomm、RackSpace、VMware 共 13 家廠商，聯合成立「核心基礎架構計畫（Core Infrastructure Initiative）」，每年資助開放原始碼數百萬美元的資金給人力、物力、財力不足的開放源碼軟體專案，甚至提供這些專案組織的主要開發人員全職工作環境，使能全心全力投入運算測試，以期對開放原始碼的安全稽核提供有力的援助。

我的網路程式設計之路走得很封閉

在我學架設伺服器設計動態網頁程式時，第一件體悟的事，不是這身技術有多棒，而是想遏阻駭客的網路攻擊是不可能的事！

所以早在 2003 年接案設計網路應用程式時，我就決定放棄網際網路的電子商務應用，改做區域網路的電腦化管理，例如工廠或辦公大樓的業務資料庫系統，把網路封閉在公司自己內部運作。否則不是增加客戶的管理成本（無論自己架站的成本或託管的每月租金），就是讓客戶因為我薄弱的技術，承擔網站無法隨時順利運作的風險與損失。

我的專業是程式設計，但網路系統管理一點也不專業，所以接案時我做了這樣的決定。

雖然如此，這樣接案有個好處。因為是自己內部的區域網路，所以不需要花錢申請網路也能跑，公司規模很小時甚至買一台分享器然後網路線拉一拉不到 2000 塊就可用一輩子（分享器不會壞的話），向工廠推銷時，特別有說服力 XDDD

網際網路的案子我只接了一個，一家披薩外送公司想要這樣的系統：「顧客在網站下訂單買披薩，然後公司的伺服器一接到訂單，會發出鈴聲通知店員，趕快做好披薩外送。」那家公司主管說好幾個月了還找不到人做出來，所以我幫忙寫程式將整套流程設計給他看。不過我沒收錢，因為我說不做網際網路的電子商務應用；想在網際網路做好電子商務生意，是要撥出成本的！沒有心專注於網路生意，純粹只是想當作額外的營運管道，甚至會變成一個坑。如果沒有心花比付給我更多的錢，給能夠管理網站讓我這套程式能順利運作下去的人，早晚會出問題要我負責或賠償，那還不如不收錢。但這案子我做得很開心，錢不是問題，能寫出程式幫助人實現想法，比拿到錢更有意義，這是錢買不到的快樂～

讓巴哈的古典音樂平衡心靈

曾經在網路廣告看到說，音樂之父巴哈的古典音樂，能夠激發左腦與右腦、均衡 EQ 與 IQ，然後幫助心靈滋長……因為這樣，我在逛書店看到有在賣巴哈的古典音樂，馬上就買回家；那時候我想像著以後聽了巴哈音樂，讓我腦波趨於穩定，工作效率就會提高，然後我網站的資料就可以源源不絕，哈哈哈哈哈。

確實聽了以後心情會寧靜下來，也覺得心情好像得到休息，這時在「絞盡腦汁」打資料的時候感覺果然舒服多了。

不過我有一個怪癖，如果一開始就播古典音樂來聽，常常聽不下去，會覺得很枯燥。還要先聽一下現代音樂例如宇多田光的 First Love 樂曲版，才能接著轉到古典音樂。

另外我對 D 調的樂曲比較有感覺，會有種醒著睡覺的效果。啥？真有這麼神奇？其實重點不在於神不神奇啦，我只是想說，沉澱心靈對寫程式或寫資料是有幫助的，做事會比較專心、能集中精神於工作項目上，而古典音樂在這方面有近乎神奇的效果，各位不妨嘗試看看。

那為什麼特別要選擇巴哈呢？例如蕭邦的音樂不是能「更快就聽得下去嗎」？個人感覺啦，巴哈的音樂聽起來比較舒服，因為他幾乎不牽動情緒地，去律動你的心靈趨於和緩，而其他古典音樂總是會有一些些些牽動情欲的感覺，讓心靈在隨著旋律沉澱的過程中稍微五味雜陳了點，沒有巴哈那麼舒暢。

不過話說回來，我手上有片現代人創作的古典音樂 CD，他則是牽動幼童般的情緒去律動心靈，其實感覺還蠻不錯的。
