《超鏈結界》序

雖然我用 JavaScript 來設計使用者介面的程式已經好幾年，並且只要可用 HTML5 平台來部署應用程式的話，就不會選擇 JVM[1]，但我照樣認為 JavaScript 是一種糟糕至極的程式語言！並沒有因為越來越慣用它，就為自己的解決方案合理化。

JavaScript 純粹是一種很有趣的程式語言而已，跳脫用傳統 Console 或 Desktop 環境來跑 Application 的慣例，把 Application 改設計成用 Webpage 的呈現方式來執行，這樣的體驗既絕妙又美好，實在是有趣極了的設計體驗。

JavaScript 也是一種很有挑戰性的程式語言，研究怎樣用這麼爛的語言[2]，寫出夠品質的程式碼[3]，一試就欲罷不能～

以 JavaScript 來驅動 HTML5 規格所架構的平台，設置出各樣程式動作，就像組裝高難度模型玩具一樣兼具挑戰性和趣味性。沒實際嘗試過的人，無法體會為何我們明知它很糟糕，卻樂於殘踏自己 XDDD

[1] 原本我總是使用 Java 程式語言設計 Application，透過 JVM 提供的 Console 或 Desktop 環境來執行；這跟改用 JavaScript 設計成在 HTML5 規格的平台跑是對等的。因此我個人將「基於 JVM 規格」與「基於 HTML5 規範」發展應用程式，視為兩者是對等的平台。
[2] 純粹只是描述 JavaScript 語言的爛，而不是反對使用 JavaScript 的爛。如果 JavaScript 真的很爛，我就不會使用與介紹它了！而且 JavaScript 原本就不是設計來發展 Web Application 用的，當初是用來讓網頁設計師能寫個短短幾行程式，把一些在客戶端就能進行的工作用網頁就執行完畢，避免不停地浪費撥接上網的時間與費用與伺服端交換訊息：「正如其名只是 script，而不是 program。」如果只是做這些工作，JavaScript 語言的設計其實很合理、方便，用起來確實有效率，並不是爛語言。爛的是後來的人犯賤，用慣了以後，硬要讓它能應付大型開發專案。
[3] 其實我還是寫不出夠品質的 JavaScript！不管我用傳統結構化程式設計的手法、或者物件導向分析設計的手法、還是特定領域框架的手法，在遇到各家瀏覽器對 ECMAScript 實作情況不一，導致計算過程不如預期時，除錯起來每種手法都有自己特定的障礙，沒有一個手法是完美的。但這就是 JavaScript 玫瑰帶刺的致命吸引力～

《超鏈結界》前言

若問及 HTML5 會是一個好的「程式平台」技術嗎？答案是讓人質疑的。

以網頁做為使用者的操作介面，然後以網際網路佈署系統，這樣的開發手法是很理想的。但是問題在於，網頁的操作介面該如何設計？

2002 年 Macromedia 提出 Rich Internet Application 的概念，輔以自家的開發環境：Flash，為網頁提供一個「使用者操作介面」的設計工具，不僅使得 Flash 這套軟體成為網頁開發者必備的標準，RIA 也成為「網頁應用軟體」這項技術的代名詞。

Flash 實現的 RIA 訴求，使得網際網路的應用相當生動，網頁更為多采多姿。至今將近十年，姑且不論 Flash 的漏洞與效率等問題，它並不是一個「開放的」或者說「標準規範的」開發平台，因為它是一款商品，你需要付錢購買這套軟體，才能用 Flash 開發 RIA。難道沒有辦法制定一個大家都能免費使用的 RIA 標準技術做為資源嗎？答案就是 HTML5！

但 HTML5 看似網頁標記語言，若要做為 RIA 的工具，其實像是結合 HTML、CSS、JavaScript 這樣錯綜複雜的一種概念與技術，你得熟悉這三種各自迥異的語法，才有辦法用 HTML5 來開發 RIA。坦白說，這與 Flash 方便的開發環境比起來，雖然免費，但門檻可高出十數倍。

在這即將改寫 RIA 歷史的時代，《超鏈結界》試著探討這樣時代的變異[1]，闡述在這在歷史動盪的時期，使用 HTML5 突破重重障礙打造 RIA 的歷程是多麼艱辛？但對時代卻有著重大的意義為何不做？

[1] 族群的變遷，往往就是時代的動盪。

《超鏈結界》簡介

雖然我說 JavaScript 是相當糟糕的程式語言，但那是因為現在我們基於 RIA 的需求與立場，來檢視 JavaScript 並不符合軟體工程生產力條件；當 JavaScript 是大型專案中的主要程式語言，將難以避免管理上的難題。

其實 JavaScript 當初是發明用來簡單存取一些頁面上的資料，然後傳遞給瀏覽器改變一下外觀狀態的程式指令集，並不是做為開發應用程式的程式語言。

早先最主要用途，是希望瀏覽器本身就能進行一些簡單的程式計算，避免像是將表單資料送到伺服端後，才告訴你資料填錯了，浪費等候網路傳資料的時間。因此 JavaScript 原本就是發明來簡單寫個幾行程式用的，並非用來寫上萬行應用程式的語言。

而為了讓不熟悉程式語言的網頁設計師，也能使用 JavaScript 進行簡單的程式演算功能，所以有寬鬆的容錯設計。

或許 JavaScript 無法就「語法」滿足大型專案的需求，必須用「哲學」的方式來實現，但做法絕對不是嚴格以 Java 這種正統程式語言的立場批判 JavaScript 該怎做，而是了解 JavaScript 最早用來簡單存取頁面資料、輕鬆改變瀏覽器外觀狀態的本質，才能在 HTML5 時代應用 JavaScript 來發展 RIA 時突破困境。

就像「無招勝有招」的哲學，思考 JavaScript 如何「簡單幾行勝上千行」的哲學，是往後能否有效使用 JavaScript 來解決案例需求的重要觀點。

《超鏈結界》導讀

《超鏈結界》的重點並非寫出符合 HTML5 規範的網頁，而是把網頁視為一種可以執行應用程式的平台，重點放在如何全程透過 JavaScript 來控制 HTML5：「甚至盡量不要用到 HTML 標籤。」所以 HTML5 和 CSS3 的部分，只關心有哪些新的特性可以展現更棒的外觀，做為網頁應用程式平台的操作介面。

然而，真正要寫 RIA 的話，建議改用 jQuery 來實作，它有效解決 JavaScript 語法不夠一致、不夠清晰的問題。不過《超鏈結界》是 JavaScript 教學單元，因此各章節依然採用 JavaScript 來實作，jQuery 的部分另闢章節來學習。

最後，《超鏈結界》示範的程式，僅相容 Mozilla Firefox、Google Chrome、Opera Browser 瀏覽器，不相容 IE8 與更舊的瀏覽器。理由除了 IE8 不支援 HTML5 和 CSS3 以外，基本上也不建議用 IE 這種笨重的平台跑 RIA；連軟體本身執行起來都很慢的東西，不需要為它寫相容性的程式。

現在選擇其它瀏覽器來使用是很簡單的事，跳過 IE 平台寫 RIA 或許才是王道[1]，不見得要浪費時間去相容 IE8。

[1] 不過 Windows 8 的 IE10 卻非常輕快，或許要對 IE 改觀了。
[*] 關於 jQuery 的部分已經過時，現在各家瀏覽器都謹守 ECMAScript 規範，JavaScript 的執行結果是一致的。而智慧型手機當道，對網路使用的流量斤斤計較，反而不建議載入越來越肥的 jQuery 了。
