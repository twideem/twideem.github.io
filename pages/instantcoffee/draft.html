多用合成、少用繼承的時機

如果只想用某物件的幾個功能，就用合成，以免繼承了其它用不到的功能。這些用不到的功能，會成為日後搞壞設計的潛在因素，這就是繼承惡名昭彰的地方。

如果你想用某個類別，但是它缺少某些功能，就用繼承的方式追加上去，然後用新的類別取代舊的類別。這種用法是安全的，濫用繼承雖然有缺點，但有它妙用的地方。

不經思考、凡事都用繼承的話，之後經常要另外寫程式防堵用不到的功能。例如你想藉由 ArrayList 的 add() 做為增加資料的功能，用繼承的話，日後很可能要防堵有人用 remove() 和 clear() 刪除資料。所以「多用合成、少用繼承」是物件導向程式設計的共同經驗法則。

最後補充一下，當你要防堵繼承而來的功能時，請馬上改用合成，而不是去防堵。

row 與 column

如果以方位來講，row 是「橫」，column 是「直」，這部分中英文是一致的。

如果以文章來講，row 是「行」，column 是「列」，但中文書寫習慣是直的為一行，英文是橫的為一行，所以這部分中英文並不一致，我們常常搞不懂 row 應該稱行才好？還是列？

要命的是，在定義數學術語時，台灣稱 row 為「列」，column 為「行」，因為「直行、橫列」，中國相反，稱 row 為「行」，column 為「列」，因為拿隊伍來講「橫排成一行、縱排是一列」，這讓中文該稱 row 是行還是列更混亂了～

在程式設計的場合，因為讀取純文字檔案裡一行資料時是橫的，而針對這筆資料剖析出好幾個欄位時，這些欄位是以直的來看，所以我習慣用中國的稱呼，row 為「行」，column 為「列」。在數學的場合，台灣人的我，就自動轉過來，稱 row 為「列」，column 為「列」。

這問題無解，寫程式要連接資料庫時 row 是行，但使用試算表時正體中文版說 row 是列～

趣談 open 與 free

Richard Stallman 不喜歡人家稱 GNU 屬於 open 精神，他認為這是 free。

這很好理解，因為毫無保留的 Richard Stallman 本來就沒有東西需要 open 出去，他成立基金會的目的，是要廠商把東西 free 出來。

但不管怎麼看，使用 GPL 一點也不自由，使用 BSD license 才是自由。

我想這是立場的問題，可能 Richard Stallman 認為程式軟體的權益遭到迫害，所以要獲得 free（釋放），怎麼會是 open（開放你來迫害我）？

用的人相反，你禁止我將修改自 GNU 軟體的原始碼以商業手法私自營用，但 BSD 可以，怎麼會是 free？明明就是 open～

我就是屬於後者「用 GNU 的人」，所以我認為 GNU 是 open，BSD 才是 free。

但我認為 open 才是真正能保障 Richard Stallman 理念的精神，坦白說我覺得 BSD 的 free 特性，其實會侵蝕 Richard Stallman 所構築的版圖，它等於吳三桂開山海關引商家入開放原始碼，讓商家在收割開放原始碼社群貢獻的結果並加以改進後，改以自己的授權條例保護起來，卻又寡不恬恥地高舉開放原始碼的大旗推廣自己的「商品」。

只有被迫 open 沒得選擇，要嘛各家廠商繼續暗自搞專利，要嘛加入開放原始碼社群貢獻，讓兩造陣營各自集中火力發展，我認為會讓開始原始碼的更健全；free 等於給了廠商「掛羊頭賣狗肉」抵銷開放原始碼訴求的可能性～

[*] BSD 的想法則是：「為何要貢獻才能使用？」不貢獻只是對既有原始碼沒有加分的效益而已，但也不扣分：「他用了並不會貢獻原始碼、不用也不會貢獻原始碼，那用不用對開放原始碼有影響嗎？」對 BSD 來說，開放原始碼仰賴志願的力量，而不是強迫的力量：「強迫只是讓不願意開放自己原始碼的人，不使用別人開放的原始碼而已。還不如開放給他用，反而有宣傳與推廣該原始碼的加分效果。」所以在這部分我觀點其實是錯的…但立場是對的，因為我是 GNU 陣營的人 XDDD

趣談開放原始碼的未來

過去個人電腦當道的時代，「應用程式」絕對是以「原始碼」設計出來的！大型成套的應用程式，則叫做「應用軟體」，雖然可能會用自己某套函式庫來設計，切割原始碼，但還是離不開使用程式語言來編譯的範疇。

但正準備進入網路平台的時代，我們發現 App 越來越像一種「客製化元件」，它只是添加網路平台所提供的互動式功能而來，不見得是用程式語言寫原始碼所產生。

未來 App 可能跟批次檔沒兩樣，只是把進階使用作業系統的一系列動作裝起來而已。即使作業系統決定要編譯才能成為 App，但我看那只是把既有 *.class 檔打包成 Executable JAR 而已，並沒有 compile 出任何東西～

往好的方面想，由於「客製化元件」的內部動作，是由前端的「網路平台（作業系統）」所提供，且目前還未聽說過有廠商能夠把「怎樣使用作業系統來做某某事」申請為專利[1]，因此「客製化元件」繼續成熟發展到完全不用程式語言來設計的話，說不定會擺脫「應用軟體」內部設計是種專利的問題，任何人都能反組譯檢視其內部結構。

往壞的方面想，提供作業平台的廠商，片面就能決定「客製化元件」的使用授權，結果 GPL 或 BSD license 再也管不到未來網路平台的元件。

[1] 目前聽過最扯的，是 Amazon 的 1-Click Ordering 專利。如果我們用 JavaScript 的 Cookie 紀錄使用者在網站上設定過的資訊，方便他下次直接點擊網站功能就好，居然就觸犯這個專利了：「那 Cookie 還能拿來幹嘛？」而 Amazon 還真的四處告設計了這樣功能的網站。幸好歐美各國後來將這專利的相關要件註銷得差不多了，雖然還是有這個專利存在，但很難拿來告人。（除非你真的從硬體到軟體的設計都照抄 1-Click Ordering 這個功能，否則只用 Cookie 保存瀏覽者資料，好讓 form 能暫時記錄他所填寫的資料這種過去便存在網頁瀏覽器的行為，已經告不起來。但除了 Amazon，其他廠商也有許多不可思議的專利，是關於 Cookie 和 Cache 的使用～）

高科技電子產品賣的是其他訴求，不意識到這點，開發不出下一代產品！

廠商以為 xPad 賣的是平板電腦，其實消費者買單的是觸控手勢。
廠商以為 xWatch 賣的是智慧手錶，其實消費者買單的是聲控智能。
同樣的，未來廠商以為 xRing 賣的是魔戒，其實消費者買單的是體感操作。
如果台灣廠商持續把重點放在開發出「更好的電腦、手機、手錶、戒指」等裝置，而不是「更好的觸控、聲控、體感」等體驗，永遠也無法領先業界推出下一代產品！

雖然消費者也覺得自己買的是平板電腦或智慧型手機，但他們為何不屑買傳統個人電腦與九宮鍵手機？存錢買平板與智慧型手機慰勞自己？因為觸控手勢帶給他們更好的體驗！所以，誰能設計出更好的觸控手勢體驗，消費者就會覺得那是比別家更棒的產品。

既然觸控手勢已經有平板電腦或智慧型手機，下一代產品就是聲控加手錶進一步簡化或輔助觸控手勢的工作，再下一代則是透過體感戒指凌空比劃幾下手勢的動作，進一步優化觸控與聲控的工作～

誰能領先業界讓聲控與體感成為高科技電子產品的絕妙操作介面，誰就暫時主宰一切，成為業界熱門焦點。

JDK 無法重新安裝

不正常移除 JDK 後，想重新安裝時，安裝程式表示已安裝過，詢問使用者是否重新安裝，確定後卻無法順利完成移除已安裝程式的動作，導致無法重新安裝。

解決辦法：

執行 regedit，依序進入……

HKEY_LOCAL_MACHINE\SOFTWARE\Classes\Installer\Products

底下有許多機碼，一個個點看看，找出 Product Name 為 Java SE Development Kit 開頭的資料，然後將整個機碼刪除。

如此就能重新安裝 JDK，而且不用重新開機。

不行的話，只好一個一個搜尋 Java 來刪除。注意！不要刪到 Microsoft 的 Java VM。

完整安裝完 JDK，總共有三套 java.exe！

在 Windows 完整安裝好 JDK，你會發現總共有三個 java.exe，例如：

C:\Program Files\Java\jdk1.6.0_45\bin\java.exe
C:\Program Files\Java\jre6\bin\java.exe
C:\Program Files\Java\jdk1.6.0_45\jre\bin\java.exe

C:\Program Files\Java\jdk1.6.0_45\bin\java.exe

雖然這個 java.exe 跟 javac.exe、jar.exe、appletviewer.exe…等各式開發工具混在一起，但這才是 JDK 真正內建來執行 Java 應用程式的環境。

C:\Program Files\Java\jdk1.6.0_45\jre\bin\java.exe

JDK 有些東西是用 Java 寫成的，為了避免使用者沒有安裝 JRE 而無法執行，因此內建一套 JRE，確保 JDK 的工具能執行。

換句話說，這並不是用來讓開發者當作測試 Java 程式用的環境，而是 JDK 給自己用的。但很容易誤會這才是 Java SDK 內建的 JRE 執行環境，以為我們應該用它來測試 Java 程式。

當然，JDK 裡面的 jre 和安裝 JDK 時接著安裝的 JRE，內容並沒有差別，是一樣的東西。所以往往會讓人覺得不需安裝 JRE，以為 JDK 裡面本來就有 jre，再裝 JRE 只是多了重複的東西。

C:\Program Files\Java\jre6\bin\java.exe

為何 JDK 的 bin 與 jre\bin 都具備執行 Java 應用程式的條件了，卻還要安裝 JRE？

只安裝 JDK 的話，Windows 並不會自動以 C:\Program Files\Java\jdk1.6.0_45\bin\java.exe 或 C:\Program Files\Java\jdk1.6.0_45\bin\java.exe 來執行 Java 應用程式，畢竟很少人會希望 SDK 同時也是執行環境[1]。

而安裝 JRE 時，會幫你配置好 Java 檔案的關聯性，讓 Windows 能自動透過 C:\Program Files\Java\jre6\bin\java.exe 執行 *.jar 檔、在瀏覽器跑 Applet、在命令提示字元呼叫 java.exe、用 Java Web Start 跑應用程式…

所以另外裝一個 Java 執行環境，對使用者與開發者雙方都是最省事、有保障的做法[2]，絕對不是裝了多餘的東西[3]！

[1] 因為 SDK 裡面通常會另外放入許多第三方套件，以滿足開發者的需求。如果這種 SDK 拿來當 Runtime 環境，測試程式時你覺得很正常，發佈出去別人常常無法執行，因為他的 Runtime 跟你不一樣。
[2] 前提是你要注意是否以 JRE 測試你的應用軟體，其實開發者幾乎都是以 JDK 的 java.exe 在測試程式！
[3] 當然，如果你安裝多個版本的 JDK，那只留最新或最舊的 JRE 就好，可以 Cancel 掉 JDK 要求安裝 JRE 的請求，其它 JRE 這時確實是多餘的～

關於 OpenJDK

本文內容已過時，僅供參考。OpenJDK 的生態不再受 Oracle 主宰後，提供的服務、以及開放的情況，都越來越好了！

OpenJDK 是 Sun Microsystems 將 Java 原始碼「部分開放」的專案，以 GPL 釋出。然而 Sun Microsystems 保留些許涉及專利的項目，並未完全開放 Java。

OpenJDK 的心態是「原始碼開放出來給專業人士得以研究 Java」，而不是「讓支持開放原始碼的人有開放的 Java 用」，因此並不積極針對不同作業系統編譯好執行檔。

到 2015 年為止，只 Linux[1] 隨時編譯好執行檔讓使用者安裝 OpenJDK，再搭配 IcedTea 做為 JRE，獲得真正開放的 Java 環境。2016 年起，Android 將改用 OpenJDK 取代 Oracle 的 Java，雖然目的是規避「Oracle 告 Google 侵犯 Java API 版權」的官司，但有望進一步活化 OpenJDK 的開放生態。

官網沒有準備可安裝於 Windows 的 OpenJDK[2]，即使找到第三方編譯好的 OpenJDK，也只是個 JDK，沒有 JRE。所以要在 Windows 推廣使用 OpenJDK 貫徹 Open 精神、打造開放的 Java 環境很困難；OpenJDK 官網甚至建議 Windows 使用者乖乖到 Oracle 下載 JRE 或 JDK，某種程度來看 OpenJDK 像是 Oracle Java 的活廣告。

版本方面，除了 OpenJDK 6 使用 JDK 7 的原始碼，然後精簡成符合 Java SE 6 的規格外[3]，其它版本皆使用相同的原始碼。例如 OpenJDK 7 使用與 JDK 7 相同的原始碼、OpenJDK 8 使用與 JDK 8 相同的原始碼、OpenJDK 9 使用與 JDK 9 相同的原始碼…

[1] 雖然泛稱 Linux，但其實只提供 dpkg 和 rpm 的安裝包，因為大多數人使用的 Linux 發行版，都是 Debian 和 Fedora 的分支。
[2] OpenJDK 官網現在有提供 Windows 的可執行檔了！
[3] 所以 OpenJDK 6 其實是 OpenJDK 7 的一個分支，而不是 JDK 6 的 Open 版。

Java 9 最大的亮點，就是新的模組化方案！

舊的模組化有許多弊端，像是以類別路徑為基礎的封裝做法，新舊模組只會彼此覆蓋，互不相容，新的模組化方案有更多細節的微調，能解決長久以來的積弊。

然而，Java 9 的模塊化功能，是屬於平台上的特性，而不是語法上的特性，它對程式設計沒有多大幫助，不要想說這是新的特性就急著導入它，應該順著未來趨勢的發展，被動地慢慢導入它。

Java 9 雖然導入新的模組化方案，但連 Java 平台本身都還未通盤地以模塊化架構規範套件，可說是在準備階段而已。因此目前還不需要去關注模塊化的議題，等到 Java 平台自己也理出頭緒規範好模塊後，我們再「被迫」照規矩導入即可。這感覺就像你不見得非要把 Java 程式封裝成 JAR 來用，但等大家都用 JAR 釋出套件時，我們就必須了解怎麼將 JAR 套件導入到程式裡面。

我建議在 Java 9 的階段，暫且不要去管模塊化這功能，搶鮮導入模塊化不會帶來助益，反而會有徒增困擾的風險。等 Java 平台的模塊化架構好了，我們再來了解怎樣切割，讓程式依賴更小的環境便能執行。目前 Java 9 可以切割的，就 Java SE、JDK、JavaFX 這幾塊，雖然對客戶端應用程式的發布還是很有用，但還是不小塊，吸引力不大。

至於為何 Java 9 給了鋸子，自己卻沒鋸好東西來，只是示範這鋸子真的能鋸東西而已？我想，或許是為了在 Java 10 或 Java 11 有新的資料型態架構後，再全新模塊化 Java 平台吧？純屬臆測就是了～
