物件導向的應用不只一種

物件導向相關書籍最大問題就是，物件導向的應用不只一種，但這些書都把「現階段」的「最後一種應用」，當成唯一、必然的一種。

我將物件導向分為五大應用層級：

１、分類 - 以結構化程式設計的思維，導入封裝的機制，更妥善模組化程式碼。
２、抽象 - 以物件導向的思維，將問題分為概念與細節兩部分來處理。
３、語意 - 從程式的再利用，進化到系統的再利用。
４、工程 - 以軟體工程的需求，在「客戶需求分析」「產品規格設計」也導入物件導向的思維，目標在建立隨時可以變動的專案管理手法。
５、生態 - 建立一個生態循環的運行環境，讓物件群以自己的方式生存在其中。

分類

一個習慣以「程式功能」來「模組化」程式設計的人，剛開始接觸物件導向程式語言，很自然會把「封裝」當作更出色的模組化手法，並且把「繼承」當作「複製貼上」來用。

這個階段的應用方式，我稱之為「分類」，特色就是「模組化」。

其實純粹以「分類」的做法來應用物件導向，自己很快就會發現物件導向應該不是這麼一回事，因為這樣做似乎跟結構化程式設計沒兩樣。

唯一感覺有比較像「物件」的，就是呼叫程式功能時比較有質感，因為好像有在操作某些東西，產生一種「觸感」，例如：

遙控器.下一個頻道();
遙控器.放大音量();
遙控器.關閉電源();
至於「多型」，可能有看沒有懂，或者看了不覺得有什麼用處，所以通常都先放著不管。

要跳脫這個應用層級，必須學習「抽象」的思維。

抽象

你講得太「抽象」了，能不能「具體」一點？

將詳細內容過程，簡單化地以某種概念性名詞來表示，就是抽象的行為。它有助於提升溝通的時效，但先決條件是對這概念達成共識。

關於抽象這一詞

台灣很少用到「抽象」這一詞[1]，如果剛好不懂英文，看到 abstract 還不知道含意，就很難體會抽象究竟是什麼東西了。

這裡摘錄《灵格斯词霸》的「維科英漢辭典」：

abstract
n. 梗概, 摘要; 抽象概念; 萃取物
v. 使抽象化, 提煉, 摘要
adj. 抽象的; 純理論的; 深奧的
看完英文的解釋還不了解沒關係，這只是幫助你大概認識抽象的詞意，往後會更進步解釋 abstract 的意義。

抽象與具象

以「日常對話」來講，沒有進一步具體去做說明的用詞，就是一種抽象的表示。例如，有人問：「外面熱不熱？」你回答：「熱。」這就是抽象，因為你不是具體去形容、或者解釋怎麼個熱法，而對方也獲得所要的回應。

反過來講，如果你具體去闡述怎麼個熱法：「攝氏 36 度。」這種做法就不是抽象，而是「具象」。在這裡你必須注意一件事！具體回答反而不見得是對方想要的答案，一個三歲小孩可能還是會問你：「那到底是熱還是不熱？」

如果你本來就不了解「抽象」是什麼意思，不妨再多花點時間，思考上述「抽象」「具象」的差異。

那程式應該怎麼寫

正統的做法是「一般化 (generalize)」。

將各個類別可以共通的部分，另外抽取出來成新的類別，就是一般化。讓各個類別繼承這一般化類別，便能以多型的機制讓各個類別的共通部分做為替換彼此的特性，例如在「操作方式 (metohd)」的參數使用一般化類別做為型態，便能讓每個繼承它的類別都可做為引數來使用，在共通的特性下，分別表現屬於自己的行為。

相對的，繼承一般化類別的各個類別，則是一種「特殊化 (specialization)」。

「一般化」就是抽象的行為，因為「共同的部分」就像一種「大家都知道的事」，所以找個語彙（抽取出來成新的類別）帶過去就好，不需要每次提及就描述細節一次。例如「今天好熱」，你不需要每次都說：「外面攝氏 40 度，平常才攝氏 28 度，今天覺得很難受。」

學術派物件導向的抽象思維

對程式設計來講，不具體去描述其功能性，就叫抽象。

說起來很容易，但實際寫程式時，根本不知道該怎麼寫才叫「不具體」。

對程式設計來講，抽象的思維應該類似這樣……

１、「汽車（car）」是具體的？還是抽象的？
２、仔細想想，汽車是由各個部位組出來，事實上根本沒有汽車這種個體存在。
３、因此「汽車」是抽象的。
４、確認抽象的是「汽車」以後，相對來講「車輪」「車殼」「引擎」「方向盤」則是具體的。
也就是說，你不用具體去描述汽車的功能性，汽車在你的程式來講，變成一種抽象的存在。

而這種「由其他事物所結合 (association) 起來的存在是種抽象」的思考方式，是跳脫「分類」的物件導向應用，進入下一階段的主要方式。

語意

一個程式語言，在使用上需要掌握的成分有：關鍵字（Keyword）、語法（Syntax）、語意（Semantic），能藉由三者表達情和理，便達到程式寫作的境地。

從人類語文的角度來看，關鍵字相當於單字，語法相當於文法，語意相當於語句。

對程式語言來講，你會用關鍵字和語法整理出程式庫之類的外部介面，將細節抽象化、概念化，但又不能失去語意，讓人誤解內部細節上的運作情況。例如你不該寫出名叫 open 的函式，卻做 close 的動作，否則儘管關鍵字和語法都符合程式語言的設計規則，你設計出來的程式沒有表達力、無法讓人類和機器自然地溝通。

題外話，讓程式語言完全變得和自然語言一樣並不是件好事，要知道無論英文、中文、日文…等等自然語言，都是資料龐大，要靠數年時間才能背熟的。程式語言應該保持簡單而精確，不是複雜而含糊。程式語言的簡單不是原罪，硬要朝自然語言的豐富努力，反而要背負巴別塔的原罪。

語意學

同一字詞，視其所應用之場合，即能明辨不同意涵，這就是「語意 (semantics)」。例如：

open the door
open your heart
open a game
因此：

A.doSomething();
或許會改成這樣：

Something.do();
程式設計的語意學，就是在探討 Something.do() 適用與否的議題。

注意「適用與否」！這表示「語意學」必須探討更多狀況，例如：

Something a = new Something();
Something b = new Something();
而這樣一來：

a.do();
b.do();
這樣還稱得上有語意嗎？顯然語意分為「字面上」與「機制上」兩種狀況。

如何透過「機制上」設計出「字面上」語意清晰的程式，讓程式碼不只是「設計功能」，而能「傳達做法」，是突破 Java 物件導向程式語言瓶頸的議題。

我們使用某個套件底下的 API 或 Framework 時，通常就會藉由語意，了解何時該用 String f1(int x)，何時該用 String f2(int x)。雖然 f1() 和 f2() 兩者做的事簡直一樣、還可以互通，但語意讓我們知道為什麼不同，而不會在 f1() 的場合用 f2()。

不同程式語言有不同語意的表達課題要去研究，例如傳統結構化程式語言要表達語意時，是在表達為什麼設計這個功能，而物件導向程式語言要表達語意時，是在表達為什麼操作這個功能。甚至同樣是物件導向程式語言，語法落伍的 Java 和語法先進的 Ruby，表達的意涵就很不一樣。

要研究 Java 的語意學，應該在「前後文的表達能力」著手，藉由更直覺、清晰的語意，降低物件之間互相搭配時的難度。

常見做法

1、名詞做為類別。
2、形容詞和副詞做為屬性。
3、動詞做為方法。
4、用繼承和合成表現句型。（A is a B、A has a B、Subject + Verb + Object）
5、對象或場合做為套件。

工程

開發軟體最困難的，是客戶需求不時在變動：「因為客戶自己也不知道要的是什麼。」而物件導向能將實際的事物對照成程式的物件，讓人感覺這樣便能夠適應種種變動，於是興起了將物件導向導入專案管理的做法。然而，並沒有物件導向管理的實際技術，只有靠經驗累積的手法，因此物件導向發展到這裡變得無所適從，成為深奧的專業領域。

其實「物件導向程式設計」原來預期的是 Framework 的開發、再利用，專案管理反而是突變狀況。

物件導向程式設計其實很簡單，讓事情變複雜的是「系統化分析與設計」，這使得物件導向被分成「物件導向分析」「物件導向設計」「物件導向編程」。

軟體工程簡介

軟體開發依工程學分為四大部分：專案、業務、製造、成品。

專案

也就是提出「企劃書」的部分。（給公司看的）

首先設定標的，然後規劃需要多少經費、配置怎樣人力、多久時間交貨，評估能獲利的話，就完成一份企劃書了。有些公司會要求企劃書說明風險，以便評估獲利的可能性。

企劃案通過，公司或開發團隊便開始向外銷售這項服務。

業務

分析「客戶需求」製作出「規格書」，做為雙方對產品認知上的契約。（給客戶看的）

這部分在軟體開發上叫做「系統分析」。但在企業經營上，通常與專案的部分合在一起，稱為「專案管理」，都由「專案經理（PM）」領導。

製造

依「規格書」製作「設計圖」，讓程式設計師實作功能。（開發團隊看的）

製作設計圖的部分在軟體開發上叫做「系統設計」，由「系統工程師（SE）」負責。實作功能的部分叫做「編程」，由「程式設計師（PG）」進行。

成品

產品交貨後的技術支援與售後服務，通常以「操作手冊」「說明書」呈現。（給客戶的員工看的）

後續維護的工作可能比前期開發的工作困難，成為軟體工程學的重大議題。

至於測試，業務階段、製造階段、成品階段，各有自己的測試計畫。

業務階段是在測試客戶滿不滿意產品的製造情況，而不是真的測試產品。因為產品設計明明沒錯，客戶不滿意也沒用，必須修改設計。產品設計得荒腔走板，客戶卻很滿意的話，反而別去動。專案經理的專業，在於說服客戶滿意，盡量避免修改設計。

製造階段的測試，是對系統運行上的內部測試。基本上在測試程式設計師們的工作結果，而不是真的測試產品，且在這樣思維下工作，壓力才不會那麼大。系統工程師的專業，在於能讓程式設計師們享受不斷堆砌小功能的成功結果，盡量避免進度壓力造成士氣低落。

成品階段的測試，是對系統操作上的外部測試。最理想的是有客服團隊專人進行這部分測試，因為這樣更能了解產品的使用情況，在面對客戶時更能解決使用者的問題。但大多沒有客服團隊，所以交由開發團隊進行。當今軟體工程的問題就出在這裡：由開發團隊面對使用者，而這不是程式設計師的專長。即使有客服團隊面對使用者，但客服卻不懂自己產品的使用情形。

系統化分析與設計

原本「物件導向程式設計」可以很單純解決「結構化程式設計」的缺點，但「程式設計」只是整個「軟體開發」的其中一部分：「製造」，因此「物件導向」未能解決軟體開發上的所有問題，甚至反而牽動其他部分製造了更多難題，特別是「專案」部分。

於是「軟體工程」為了因應導入「物件導向程式設計」後所牽動的部分，「專案管理」這個部分也開始「物件導向化」，物件導向變複雜的原因就在於此，分為物件導向分析、設計、編程。

由於「專案小組」指揮「製造團隊」「產品人員」應該怎樣做事，當專案小組的「產品規格設計」也物件導向化後，物件導向程式設計師的工作變成直接按照這規格把功能實作出來就好。這是一個很理想的做法，程式設計師必須遵照上層統一規定的「介面」將功能實作出來，但程式設計師依然可以依照自己實際遇到的情況以各式「類別」來實作。

前提是專案小組的產品規格設計，必須以正確的物件導向手法來規劃，才能架構出可供程式設計師盡情「分類」的「介面」，否則程式設計師將無法在「介面」上多做改變，淪為只能依照「介面」輸入動作內容的「打字員」。

然而並沒有一個嚴格純正的學術理論定義物件導向專案管理必須怎樣，因此物件導向的專案管理手法需要相當程度的技倆，這使得物件導向的專案管理手法相當仰賴經驗的累積，並非只靠學術上的研究就能掌握。

現階段物件導向手法的潛在問題

物件導向的精神是「抽象」，而不是「想像」，但程式設計師真正抽象的對象其實就是「程式」本身，所以一個成熟的物件導向程式設計師，是以「讓程式去像事物」的手法來編寫程式。

當程式設計師必須遵照「產品規格設計」寫物件導向的程式時，就變成一種想像，因為又回到「讓現實的事物來像程式」的思維，而這是錯誤的抽象行為。

所以當專案小組以不正確的物件導向手法為產品規格設計制訂介面時，物件導向程式設計師很容易被限制分類的做法，純粹只是將功能實作出來而已。

當然，這並不影響整個軟體開發的進行，所以不曾有人正視這個情況，加上這樣的開發手法已被證實能提高軟體開發的時效管理：「特別是能隨著客戶需求的善變直接更動設計，也不破壞進行中的開發過程。」因此你也不需要去重視，除非你想從「學術上」繼續研究物件導向，才需要了解這並不是物件導向程式設計該有的樣貌。

現階段物件導向手法的改進空間

物件導向專案手法已經完美，但就軟體開發而言依然不夠完美。

物件導向的專案開發手法，只是方便管理，並不代表那就是真正「物件導向程式設計」該有的樣貌。但物件導向的專案管理手法已經成熟且定型，只能說程式設計員已確定是勞工階級的職業，若想晉身管理階層，應該熟悉那「所謂真正的物件導向」。

最後透露一點，成功導入物件導向專案手法之後，大家認為「已經完全物件導向了！現在用的是最終形態的開發手法。」其實不然，因為軟體工程分四大部分，而現在只有業務和製造兩部分物件導向化而已。想完全實現物件導向的軟體工程，必須等到「成品」執行於「物件導向的作業系統」（現在是程序型的作業系統），也就是程式不再只對程式自己以物件導向的脈絡來傳遞資訊，而是作業系統本身就能以物件導向的資源讓程式以物件的方式來溝通，這才叫最終極的「物件導向軟體」。[3]

物件導向的開發手法會因此再突破性成長一次，並達到真正該有的時效性。「軟體」將漸漸以「系統」的型態存在，因為這種作業系統能直接讓使用者從各種系統功能中，抽出自己想要的部分，來組合成一個軟體；使用者該做的事只是佈置一張操作介面，一一連結想要的這些功能而已。設計軟體變得沒有門檻，只剩有多少系統功能可用而已。

另外，現行的「物件導向應用程式伺服器」，例如 WebSphere Application Server 或 JBoss 之類，跑的是「系統」，而不是「軟體」[4]，所以現在還不能幼稚的認為軟體工程可以完全物件導向了。

結語

現階段物件導向發展到「工程」這個應用層級，也就是將物件導向的應用跨展到整個軟體工程的專案管理。這確實是普遍的走向，但絕對不是唯一的走向，因為程式設計的目的並不只是開發大型專案[5]，若認為物件導向就是應該如此，而否決掉其它應用手法的存在價值，反倒是偏廢的行為。

甚至，有些情況不導入軟體工程反而更容易維護。怎麼說？導入專案管理，其實等於買保險一樣：「犧牲些許成本，保障主要利益。」意即它並沒有更方便維護、也沒有更提昇開發效率，反而是犧牲些許方便與效率，但規劃出保證可以維護、能夠掌握時程的開發效率的產線。

物件導向程式設計的好與壞，在於事前犧牲寫程式碼的效率，日後換來擴充與維護的方便。

然而隨著物件導向五大應用層級的提升，會漸漸發現兩者並不衝突、或者說本來就是一起進行的。

物件導向只是要求你寫程式時兼顧系統化分析與設計，就像小說家寫作時要架構故事一樣，寫歸寫、故事歸故事，兩者是並進的，寫程式前系統化分析與設計，系統化分析與設計後寫程式，其實也是可以同時進行、不衝突的。關鍵就在你是否以小說家的思路寫小說、是否以系統工程師的思路寫程式，設計的思路對了，你只是在系統化的同時寫上程式而已，就像在物件導向程式語言的殼子裡面寫上結構化程式設計，基本上沒什麼變化，比較像進化，從結構化程式設計進化到物件導向程式設計，而不是從結構化程式設計變成物件導向程式設計。

也因為這個原因，沒這思路，你寫出來的物件導向程式，其實就跟傳統結構化程式設計沒兩樣，看不出哪裡換來日後的擴充與維護，卻犧牲了寫程式碼的效率～
