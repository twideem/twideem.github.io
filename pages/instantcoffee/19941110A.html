物件導向設計模式

首先，「物件導向設計模式」並非將「物件導向程式設計（OOP）」中反覆出現的「程式結構」整理出來成「名目表」，而是將「物件導向設計（OOD）」中屢次有效解決開發需求的實作方式整理出來。因此「程式的結構」並非「物件導向設計模式」的重點，「需求的解法」才是重點。比如說，看到 Facade 表示把複雜的系統簡化成容易操作的介面來使用，看到 Adapter 表示將別處不相容的操作介面轉換為相容此處的操作介面來使用，兩者所透露出來的設計理念才是重點，至於實作出來以後程式碼長怎樣並不是重點[1]。

但程式碼確實會說話，對於沒有物件導向分析與設計經驗、或者用 UML 展示物件導向設計模式，反而無法在第一時間領會其奧妙的人[2]，從程式碼的結構來了解設計模式，多少還是有幫助的。因此，本文使用精簡而容易閱讀的程式碼來示範設計模式，讓習慣用程式碼探究原理的人更輕易領會。

為了方便研究程式碼的結構，可下載本單元所有範例：19941110A.7z，一邊操作其行為結果，一邊探究其組織方式。

目錄

1994 年出版了 Erich Gamma、Richard Helm、Ralph Johnson、John Vlissides 四人著作的經典鉅作《Design Patterns: Elements of Reusable Object-Oriented Software》。這本書之所以讓人讚嘆不已，在於它不只是「電腦圖書」，更是淬鍊人類智慧的「學術著作」。我們在藉由這本書學習到「物件導向設計模式」之餘，更目睹這「四人幫」(GoF: Gang of Four) 是如何展現出人類的學術潛能，能把「設計模式」寫得透徹、而且分明。有程式設計背景的人，讀這本書保證像在讀一本文學著作一樣，會讓你見識到原來電腦書籍還是可以寫得經典，由衷對整本書的安排讚嘆不已。

言歸正傳，要建立「物件導向開發手法」卻苦於沒有經驗的累積，那最有效的方式，就是接受「物件導向設計模式」的歷練！這些設計模式是將「物件導向程式設計」十數年來，反覆出現的物件組合方式[3] 一一整理出來，讓世人更清楚了解：「物件導向平常是怎樣被使用的？」

補充說明

本文盡量不對程式碼作說明，希望讓讀者自己循程式碼的脈絡，看看究竟發生什麼事？這樣才能體會每個物件導向設計模式的靈魂所在。其實，如果反過來，變成讓你照我的說明比對照程式碼：「看是不是如我所說的這麼回事」，那你了解的可能是我說明的意義，而不是程式碼本身的意義。或者說，當你了解程式碼本身的意義，你會做出跟我不一樣的說明，因為每個人欠缺與關心的地方不同，需要的說明不一樣。

也因此程式碼盡可能寫到最短，讓讀者能在五分鐘之內搞清楚在做甚麼。

本文的缺點是，因為過於簡略，只把每個設計模式最關鍵的結構成分表現出來，所以並不符合原著《Design Patterns: Elements of Reusable Object-Oriented Software》真正的設計模式精神！原著在介紹一個設計模式時，通常該設計模式包含了其它的設計模式，讓這個設計模式可以跟其他設計模式銜接，也就是說，原著的精神是「軟體開發的可再用性」，並不是程式碼的結構本身。例如 Bridge 模式，符合原著設計精神的話應該是這樣：

java.lang.Object.clone() 可用來 Prototype，畢竟只是複製一份物件做為新的物件。
java.lang.Runtime 就是以 Singleton 實作。
java.awt.event 裡面有許多設計為 Adapter 的類別。
javax.swing 的元件往往有個 UI 屬性，這就是一種 Bridge 的設計。
java.awt.Component 是一種 Composite 的表現。
java.io 許多 Stream 系列，就是 Decorator 的很好案例。
java.lang.String.intern() 就是 Flyweight 的表現。
java.lang.reflect 是 Java 內建的動態 Proxy 模式。
java.util.Iterator 是 Java 內建的 Iterator 模式。
java.lang.Observer 是 Java 內建的 Observer 設計模式。
java.awt 中 Container 以 LayoutManager 切換排版規則，就是一種 Strategy 的表現。
java.awt.Component 中一系列 process 方法，很符合 Visitor 模式的精神。
