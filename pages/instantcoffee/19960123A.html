如何學習 Java 程式設計

學習 Java 程式設計的幾個要點如下：

１、了解自己選擇 Java 的理由，但告訴自己 Java 不是最好的。
２、打造 Java 環境，並練習最主要的操作方法。
３、熟練 Java 語法，但別耽誤太多時間在這上面，其實語法不是程式設計的重點。
４、明白電腦運作原理，能讓你思考程式設計時，清楚自己在做什麼。
５、了解物件導向有他簡單的一面，然後建立一套自己的開發手法。
６、從軟體工程的系統化分析與設計了解物件導向複雜的一面，來掌握物件導向的精隨。
７、務必學會查閱 Java API Specification 文件來設計程式。
８、成熟的程式設計師，不在於技術，而在能以平常心除錯。
９、效法外科醫學院生，每項手術求熟練，不求速成與貪多。
１０、未來還有很多需要延伸的議題可以學。

以下將針對各點提出指引：而不是答案。因為你必須花時間找出自己的答案，才會對你的學習帶來幫助。每個要點可能會花你兩三天時間找到答案，但絕對比我把答案貼上來，讓你只花十分鐘就有答案更有幫助。

了解自己選擇 Java 的理由，但告訴自己 Java 不是最好的。

這樣才不會自己騙自己，而能冷靜分析癥結點在哪，然後懂得做適當的取捨。程式設計過程中，不懂得捨棄不適合的項目，硬迷信自己選擇的是萬靈丹，結果產生出來的程式往往都在破壞原則，搞爛掉整個設計。

其實不只 Java，學任何程式語言也都應該如此。

就算每個階段都有最好的程式語言，但如果你 BASIC 用沒一年換 C，沒一年又換 Java，沒一年又換 C#，沒一年又換 Ruby，我看你能做的事情可能比 BASIC 用四年的人還少。最好的程式語言不等於完美的程式語言，而真正接近完美的其實是長期對某一語言之缺陷所累積的解決經驗。

覺得語言不夠用，所以多學其他領域的程式語言則另當別論，例如 BASIC 無法滿足網頁程式設計需求，所以多學 JavaScript，然後又學 PHP，這種是各種領域都學一種語言，而不是同樣一個領域不停換新的語言。

初學程式設計，一種語言請起碼堅持三年的使用經驗，才會帶給你學習效果。

打造 Java 環境，並練習最主要的操作方法。

想開發 Java 程式，就安裝 JDK。開發出來的 Java 程式想拿給人家執行，就安裝 JRE。

最主要的操作方法是 javac 與 java 兩個指令的用法，這部分請加強練習。

不妨上網搜尋 javac 和 java 的相關資料，然後解釋下面兩道指令在做甚麼：

javac

javac -classpath bin -d bin -encoding UTF-8 src/*.java

java

java -classpath bin application.Main

剛開始學 Java，最讓人困惑的，就是編譯出來的程式不像執行檔該有的樣子，搞不清楚該怎麼執行。

沒錯！坦白說，我們編譯出來的程式，真的不是執行檔。執行檔是 JRE 為 Windows 作業系統提供的 java.exe 才對，我們寫出來的 Java 程式只能說是 java.exe 這個執行檔的外掛！

了解這點，然後接受這樣一個事實，就不會困惑了～

熟練 Java 語法，但別耽誤太多時間在這上面，其實語法不是程式設計的重點。

以結構化程式設計的觀點（而不是物件導向的觀點），熟練「變數」「型態」「陣列」「條件」「迴圈」「函式」等語法；能說明這些東西的用途即可，不用過度深入研究。很多人誤以為這些就是程式設計，而準備花一整個月、甚至整個學期去學習[1]。其實這些都只是「語法」，是相當簡單的東西，會用就好，根本不需要花太多時間在上面。

想花時間練習是因為「語法」被誤會成「邏輯」，不過「邏輯」不是靠練習「語法」能有效獲得的，而是在實際嘗試寫軟體時才能有效獲得。

所以別耽誤太多時間在語法上面，一天專心練習其中一個語法，一星期就能練完這些基本的語法。然後開始嘗試寫軟體，想辦法用這些語法展現你心目中想要的功能，自然會發現到有好多種「邏輯」可以嘗試其演算結果。

另外！當你摸索邏輯時感到迷失，由上而下按照順序執行，就是最棒的邏輯之一。

明白電腦運作原理，能讓你思考程式設計時，清楚自己在做什麼。

了解 CPU 與 RAM 是怎麼搭配起來運作的，這不僅是電腦運作的基本原理，更是程式實際執行的方式，所以能讓你知道程式設計是什麼東西，清楚自己透過程式設計應該讓電腦怎樣動作，從此對程式語言就得心應手了。

此外也應該思考 HD 與 OS 在這運作原理上對程式設計帶來的變化，程式語言演變成讓作業系統動作，而不是直接對電腦下達命令。不過這並不影響最原先的運作原理，只要知道有這種狀況就好。

CPU 演進成微處理器的架構後，BUS 對整個電腦運作原理也是很重要的議題，但現在學的是 Java，不是組合語言，所以基本電算原理以外的複雜主機板架構，對應用軟體的程式設計師而言可以忽略。

了解物件導向有他簡單的一面，然後建立一套自己的開發手法。

先了解結構化程式設計將程式分成三種結構：

．循序結構
．條件結構
．重覆結構
且暫時不要一遇到可獨立性的程式區塊就使用函式，而是為了讓結構化程式設計表現得更清楚才使用函式，以做為你的基礎開發手法。

熟練了以後，再加入物件導向程式設計來「分類」你的結構化程式設計，並了解這樣的程式最主要差異在於「資料」的獨立性，能幫助你「擴充」程式功能，即可展開你的物件導向開發手法。不過這時你應該思考物件導向的「類別」是如何破壞結構化程式設計的「禁止 GOTO」原則，然後試著將這種破壞表現為程式功能在擴充時才會有「彈性」的手法。（結構化的流程像序列，物件導向的流程像並列，應用得宜就能避開 GOTO 的缺點。）

不知道如何嘗試的話，不妨將函式全部改成類別來寫程式，以這種寫法試著開發小型軟體或簡單的工具，很快就能見識到物件導向的特點。

至於「繼承」與「多型」，等你需要開發「容易就能變更設計（而不是修改程式碼）」的程式再來研究。

補充題外話，Java 自稱完全物件導向的程式語言，顯然就語法來看是個謊言。但如果將 Java 的「動態載入」視為物件導向的特性之一，確實可以實現完全物件導向的程式設計。這點將對你的物件導向手法帶來很大幫助，請務必思考這樣的特性。

從軟體工程的系統化分析與設計了解物件導向複雜的一面，來掌握物件導向的精隨。

原本「物件導向程式設計」可以很單純解決「結構化程式設計」的缺點，但「程式設計」只是整個「軟體開發」的其中一部分，因此「物件導向」未能解決軟體開發上的所有問題，甚至反而牽動其他部分製造了更多難題。

為了因應導入「物件導向程式設計」後所牽動的部分，「軟體工程學」的「專案管理」這個部分中，「客戶需求分析」「產品規格設計」也開始「物件導向化」，物件導向因此變得複雜，分為：物件導向分析、物件導向設計、物件導向編程。朝這方向發展，你就能了解物件導向複雜化的原因。

但是，物件導向的專案管理手法是需要很多技倆的，且必須從專案管理的過程中，學習如何導入物件導向的手法，而不是研究物件導向就知道如何導入專案管理。因此這需要經驗的累積，而不是只靠學習而來，所以物件導向的專案管理手法是有門檻的。

好消息是，在物件導向的專案管理手法下，程式設計師只是遵照「產品規格設計」所制定的介面將功能實作出來的「打字員」而已，這意味著只要學會這有門檻的管理技倆，你就是有管理能力的人才。

物件導向的專案管理手法其實是一種技倆，這種物件導向的發展，目的是為了方便管理而已，並不是物件導向程式設計該有的樣貌。然而它確實能有效提高軟體開發的時效，因此你不需要去正視這個議題，當作茶餘飯後的話題就好。

務必學會查閱 Java API Specification 文件來設計程式。

開始學 Java 程式語言，你首先要掌握的是傳統結構化程式設計的語法如何靈活應用，接著要了解物件導向程式語言有哪些組織程式的方式，然後就是研究 JDK 提供那些 API 供我們使用。所以，務必學會查閱 Java API Specification 來使用各個類別的功能！如果你看不懂且不會用 Java API Specification，保證是鱉腳的 Java 程式設計師！無論如何都要學會才行！

所謂 API 就是類似「程式庫」的東西，由對方預先幫我們實作出各式各樣的程式功能，然後我們只需要呼叫這些功能來用即可。

透過 API 來見識 Java 是怎樣應用物件導向，是一定要有的經驗，否則無法了解 Java 物件導向程式設計是怎麼回事。但凡事總有個順序，我建議先用 Collections 這套 API 來認識比較好。先學會用群集來寫程式，讓程式的主體是以這些群集所構成，然後深入研究 String 與 StringBuilder 兩個物件為何不同，而懂得避免製造垃圾的話，剛開始就能寫出漂亮的物件導向程式設計[2]。

在你還沒有程式設計的底子之前，很多 API 或者議題，會讓你彷彿陷入謎團之中，影響你觀念的建立，而干擾學習進度。這時，屬於資料結構與演算法的 API 務必熟練，其他的 API 把練習結果保存起來即可。

成熟的程式設計師，不在於技術，而在能以平常心除錯。

初學程式設計的人，想達到每次程式寫出來保證都能通過編譯的境界，在於除錯經驗的累積，好讓你懂得規避錯誤。絕對不是因為在學習時，靠著聰明才智而充分理解語法，所以能寫出正確無誤的程式。

因此，千萬不要在寫的程式總是編譯錯誤時，自認為沒天分而感到氣餒，那是靠除錯經驗而來的。以平常心面對錯誤，心平氣和地逐步解決，才是成熟的程式設計師。

況且，很多錯誤並不是語法上的錯誤，而是邏輯上的錯誤，軟體執行時才發現運算情況不對。這事很常發生，想完全不除錯不太可能，所以一名成熟的程式設計師是不會對除錯感到意外，只會很稀鬆平常的接手處理。

甚至高竿的程式設計師，會故意寫錯誤的程式碼讓編譯器咆哮，以確保程式的品質。所以不要把編譯錯誤的訊息當作責備感到挫折，而要把會發出錯誤訊息的編譯器當聊天對象；日後你還會嫌它話太少、口風太緊，打探不出你想聽的八卦 XDDD

效法外科醫學院生，每項手術求熟練，不求速成與貪多。

物件導向程式設計的學習，不能為了貪多而求速成。

應該效法外科醫師，在醫學院學習新一項手術時，就專注研究，反覆練習，直到全盤熟練為止[3]。寧可穩紮穩打只熟練幾項手術，也不能一知半解的情況下快速學遍一大堆手術。

事實上，每個項目經過充分練習，自然會研究出箇中道理，看似裹足不前，但學習的成效反而比貪多速成來得快。如果為了搶快貪多，沒經過多少練習就往下一個項目學習，到時候自以為學完整個課程，其實並未建立起多少足以落實的伎倆與心得，結果空有觀念卻無法應變，等於沒學通，回過頭來重新學起又覺得不自在，耽擱下來反而是最慢的學習方式。

與第三項要點相比，第九項要點講的是「物件導向」要多花時間去練習，「語法」則是不要過度研究箇中道理，因為實際應用它自然能體會出來。看似相反，其實理念一致：「多實際練習，不要只在觀念上學習。」

「語法」指「程序導向」那些傳統語法，學了以後應該一次應用到所開發的程式案例上，直接就當作練習。「物件導向」的那些特殊機制，則應該熟練到充分了解其意義了，才應用到程式案例裡面。

未來還有很多需要延伸的議題可以學。

現在很多程式設計師，其實是熟悉某工具軟體，並不是真的對於程式語言有過人的能力，這樣跟其他熟悉製圖軟體、排版軟體、文書軟體的人其實沒甚麼兩樣。

電腦只是輔助工具，除了熟悉工具軟體的操作方式，還應該具備工具軟體背後的專業知識，才會有競爭力。如果你只會 Visual Basic，那麼在電腦技能的專業程度上，其實不如熟悉 Photoshop 且擁有平面設計專業知識的人。

這時你可以選擇專精於「伺服端程式設計」「遊戲程式設計」「軟體工程」來發展，或者涉獵「資料結構與演算法」「物件導向設計模式」「安全性設計（加密協定、通訊憑證）」……來強化你的程式技術。

光會使用「程式語言」本身，只能說是「使用者」，稱不上「設計師」和「工程師」，並不具備競爭力！

結語

在學會用 Java 語言來開發應用軟體，而覺得自己算是跨入「程式設計」的領域後，你會發現，Java 要學的東西特別多！而且好像沒學起來的話，就不算是 Java 開發人員，像是：Java EE、Servlet、Struts、Sprint、Hibernate…

的確，那些 Java 主流技術沒摸透的話，稱不上「Java 開發人員」，但並不表示你非得學起來不可，因為你可能只是想當個「軟體開發人員」，而不是「系統開發人員」，這樣的話，那些東西即使沒去接觸，也沒什麼影響。

雖然 Java 擁有簡單易用的 GUI 套件，但是在開發 Application 的情況上，依然不受青睞；大家還是喜歡用 C 語言寫應用程式、桌面軟體，Java 則用在企業系統的場合。不過…

只想拿 Java 來開發 Application 沒什麼不好。

甚至我鼓勵 Java 初學者，先專心在 Application 領域奠定基礎，當你真的從 Desktop 跨入 Server 領域了，再去研究有哪些 Framework 可以讓系統架構設計出來更容易維護。不要為了學不會那些與 Application 不相干的東西，以為自己不適合學程式設計。

當你為 Java 許多在網路上蔚為主流的 Framework 感到疑惑不安時，可以問自己：「我是為了學會 Java 而學程式設計？還是為了學會程式設計而學 Java？」
