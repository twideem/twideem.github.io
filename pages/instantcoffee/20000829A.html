物件導向開發的癥結點

在《物件導向的應用不只一種》洋洋灑灑提了一堆，恐怕未能點出物件導向開發的癥結點，因此現在直接針對問題做出探討。

物件導向程式設計的癥結點

現實世界的事物，不可能全部搬到程式裡面設計。電腦只是人類的輔助工具，亦即在現實世界裡，它只分擔部分工作，其他部分依然由人類自行處理。因此以為物件導向程式設計連人類自行處理的部分都要設計進去，是開發手法感覺起來不倫不類的原因。

來看看電腦的使用過程，這也是你熟悉的傳統程序導向程式設計思維：

人工作業輸入→電腦程式運算→裝置結果輸出

跟程序導向一樣，只是「電腦程式運算」這個部分，物件導向提供更強大的程式管理機制，讓你更能夠維護軟體的開發。但僅止於此，程式設計依然是程式設計，不是涵蓋程式設計「以外」的工作，因此回歸電腦本身才是正途，不要妄想把現實世界的一切：「人工作業輸入」「裝置結果輸出」也搬到電腦裡面。

所以物件導向開發手法，依然是以「操作功能」為主體，不要以「實體對象」為設計的主軸。

把傳統開發手法寫出來的程式結構，進一步用物件導向的語法來組織。就這樣而已，不需要用想像力去思維程式與現實世界的對照關係。

Java API 就是最好的例子，請問裡面有 Cap（帽）、Car（車）、Cat（貓）這種現實世界對照而來的類別嗎？沒有，都是針對「程式」本身的「結構」，用物件導向的機制重新規劃成類別，讓開發者用「組成」的方式來完成一套軟體而已。

所以你應用物件導向開發程式時，也是一樣的手法，針對你的程式本身做出規劃即可，根本不需要什麼所謂的「物件導向思維」去扭曲你的程式。

物件導向專案管理手法的重要性

雖說「物件導向」變成一種「軟體工程學」是突變的發展，但這只表示並非「必然性」，而不是不具「重要性」。在這客戶需求不停變動的產業環境，物件導向專案管理手法是很重要的！

開發軟體時，程式語言技術本身通常不是問題。在軟體真正的模樣呈現出來以前，其實客戶（可能是自己的老闆）自己也不知道想要的是什麼東西，看到以後才挑三揀四說跟他「所想要」的不一樣[1]，才是困擾軟體開發的天大問題。

所以藉由物件導向把現實世界搬到程式環境裡面，讓我們在這總是被迫來回更動設計的開發工作過程，能有個直接的對照關係來迅速分析「可以調整的環節有哪些？」是非常有意義的專案管理手法。但光憑「封裝、繼承、多型」的物件導向程式語言，其實不可能達成這個目的，導致我們變成運用想像力在管理專案罷了！

封裝、繼承、多型的用處，是能將開發工作切割成一塊一塊的，分配給程式設計師進行。誰趕不出來、或誰的出錯，並不影響其他程式設計師的工作結果。過去以功能為主的傳統結構化程式設計，每個人的工作結果難免會牽連到他人的工作進度，想開發的系統只能架構成一道道關卡通行而來，一個環節錯了，整個系統就無法動彈。借助物件導向能將程式分割成各自獨立、互不牽連的個體，才能將想開發的系統，架構成由一個個已經做好的組件堆砌而成，完成多少動多少，還沒做出來的不用擔心影響他人，成為系統工程師最強大利器。

物件導向程式語言不能解決軟體開發的問題，但在期待下一型程式語言出現以前，還是只能仰賴物件導向專案管理手法來解決軟體開發的問題。或者，建議依賴 UML 做為專案管理語言（或稱工具），用來把程式語言本身給虛化掉、不要讓軟體設計直接受程式語言影響，就能解決「物件導向程式語言」與「專案管理手法」不妥協的問題，是目前最棒的軟體開發方式[2]。

採用物件導向的開發效益

用物件導向程式設計來開發軟體或系統，其實撰寫程式的工作速度會變慢，專案進度更不會變快。採用物件導向的好處，純粹是維護和擴充時，要比傳統程式設計來得快。也就是說，導入物件導向來開發大型軟體系統，等於放棄前期的工作效率，來換後期的維護效率。

那為什麼要採用物件導向呢？因為專案管理最常出現且棘手的問題，不是程式寫不出來，而是難以變更規格與修改實作，所以導入物件導向是加分的！

然而，大多初學 C++ 或 Java 的人，並沒有認知到這件事，不知道自己寫程式、寫軟體的速度不快[3]。也不知道自己既然用物件導向程式語言了，就該寫出易維護的程式，而不是很快地把程式寫出來。於是物件導向怎麼學也學不出所以然來，寫了三年的 C++ 或 Java，雖然會覺得自己很會寫程式，但對於物件導向的開發手法是什麼，幾乎都是一頭霧水。

沒有正確地認知到物件導向是門犧牲前期工作效率來換後期維護效率的技術，甚至錯誤地以為物件導向會讓程式設計寫得更快，也是讓你無法建立物件導向開發手法的癥結點之一。

題外話

既然物件導向寫程式的效率低，初學程式設計的人，不見得要從物件導向程式語言開始學起。如果你只想自己一人寫簡單的應用程式軟體，沒想過參與多人團隊，共同開發大型專案，那學 PASCAL 或 C 語言這種傳統結構化程式語言也無妨，不需要因為 Java 是最流行的程式語言，就覺得學這個比較好。因為易維護的好處你可能享受不到，寫作慢的壞處倒是逃不掉。

然而，現在學古老的 C 或 PASCAL 寫程式，難免受人鄙夷，畢竟大家都是盲從之輩，看到誰沒跟著主流走就覺得落伍。不想忍受大夥兒說三道四，就選 Python 吧！它只是支援物件導向程式設計，並不強迫非得物件導向才能寫程式，寫作速度快得很。

當然，也不是說 Java 不好，Java 有史詩級壯闊的 API 可用，以彌補物件導向程式設計寫作慢的缺點[4]。我的意思是不要迷信 Java，如果你學物件導向覺得很沒趣，好像怎麼寫都不夠物件導向，覺得想改 PASCAL 的傳統程式設計，把一行行把程式碼敲出來就很結構化，好讓你把重點放在寫程式本身，不想為理論上的東西疑東疑西的，那隨時掉頭無所謂，不要在意別人的看法。

除非你學程式設計是為了職場規劃，而不是個性使然，那就真的要照潮流走了～
