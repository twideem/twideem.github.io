再見了，Java Applet～

Java Applet 還值得做些什麼？請看看 JApplet，它只需寫一小段 Java 基本 API，就輕易在網頁上執行視窗程式 …｛但是｜應用程式已被安全設定值封鎖｝（Firefox 52 起不再支援 Applet，想回味過去寫的 Applet，可用 appletviewer。）

re:Java Applet 還值得做些什麼？

雖然 Java Applet 很迷人，例如能在網頁開 Metal Look and Feel 風格的視窗來跑，用這種方式打造應用程式很酷！但 Applet 限制很多，對開發者來說綁手綁腳，只適合單純執行 Java 程序，不適合對外存取想要的資源。想用 Java Applet 設計 RIA 變成自虐的行為，選擇 Adobe Flash 或 HTML5 會比較輕鬆。

沒想到打入冷宮的 Applet，變成駭客研究後門掌控權限的管道，使得 Java SE 7 連番爆出怎麼補也補不完的安全漏洞，補到後來 Applet 限制越來越多，越來越難用，實用度大打折扣。限制過苛的 Applet 現在能做的事比 HTML5 還少，變得快沒有存在價值，差不多是退場的時候！

說真的，建議把 Applet 從 Java SE 抽掉，變成另外下載的套件選項，有需要的人再額外安裝會比較妥當。順便降低 JRE 檔案大小，尤其 JRE 越來越肥了，從 JRE 6 的 17MB 暴肥到 JRE 7 的 30MB。（64 位元的 JRE 8 又暴肥到 60MB）

原本應該放寬限制，提升 Applet 實用性來增加使用率以延續壽命。但即便如此，將 Applet 從 Java SE 抽出來，以擴充功能的方式做為進階使用，也是相當合宜的做法。

[*] 真的抽掉 Java Applet 了，請參考《Oracle 不再支援 Java 瀏覽器外掛》。

根本補不好的 JRE7 零時差漏洞攻擊

成為黑客基本攻擊工具的，只發生在 JRE7 的版本，且透過瀏覽器來鑽漏洞，因此建議不需要在瀏覽器執行 Java 的話，將該功能關閉，不然就是改用 JRE6 比較安全，因為至少 JRE6 的漏洞不至於成為主要攻擊對象。

Java SE 7 Update 51 的 Applet 必須強制簽署

前言

自 2012 年 8 月 30 日 Oracle 緊急釋出 Java SE 7 Update 7 修補零時差攻擊的漏洞開始，到 2013 年 9 月 10 日的 Java SE Update 40 總共經歷 10 次的更新，Applet 的漏洞依然補不完！

結果 2013 年 Java 成為網路攻擊最主要標的，大家開始發起移除 Java 的運動，徹底砸了 Java 這塊招牌[1]。

因此 Oracle 到了 Java SE 7 Update 51 起，乾脆強制規定所有 Applet 程式都必須通過簽署，如果沒有簽署、或者有簽署但已經過期，一律不允許執行。

然而 Oracle 把自己無能為力的責任，轉嫁給使用者承擔的做法，反而讓想跑某 Applet 程式才安裝 Java 的使用者移除軟體！因為，單純 Applet 的使用者、而非 JDK 的開發人員，大多不曉得有「Java 控制面板」可以設定白名單，執行未簽署的 Applet，以為 Java SE 7 再也無法執行 Applet 後，乾脆移除算了。

做法

以 Windows 8 為例，在「控制台」→「程式集」裡面，可以找到「Java (32 位元)」，這就是「Java 控制面板」。

在「Java 控制面板」的「安全」頁籤裡面，可以透過「編輯網站清單」按鈕，輸入 Applet 程式的網址，就能避開強制通過簽署才能執行的規定。

感言

Applet 的漏洞破壞 Java 形象是一回事，Oracle 造成使用者不便讓人乾脆移除 Java 又是一回事；無奈傷害已經造成，2013 年讓 Java 一世英名毀於一旦。

期望 2014 年的 Java SE 8 能展現新氣象，讓我們這群 Java 開發人員能重新向使用者推廣安裝 JRE 的好處…現在很難想像這是多麼艱鉅的任務。

[1] 我們標榜 Java 寫的程式跑在 VM 所以「很安全」，現在裝 VM 等於開放漏洞，安全牌沒了。

應用程式已被安全設定值封鎖

由於 JRE 7 開始，漏洞一堆的 Applet 成為網路駭客攻擊最主要目標，而且怎麼修也補不完，最後 Oracle 決定只有申請安全驗證的 Applet 才能執行！

因此，如果你願意執行本站 Applet 範例程式的話，建議將網址加入到 Java 的「白名單」：

「控制台」→「程式集」→「Java (32 位元)」→「安全」→「編輯網站清單」→「新增」→「https://twideem.github.com」→「確定」。

重新啟動瀏覽器，再點看看，應該就能執行。

再不行，只能關閉驗證功能或調降安全層次，但不建議你這麼做！寧可保持 Java 的系統安全預設值，也不要為了跑本站的無聊範例去關閉它～

而且「Java 控制面板」裡的「進階」，並沒有還原預設值的按鈕，改完後悔想還原，通常已經忘記修改前的值是什麼，結果改不回原貌～

Oracle 不再支援 Java 瀏覽器外掛

Java 官方 Blog 發布《Moving to a Plugin-Free Web》，未來 JRE 不再提供瀏覽器執行 Java Applet 的外掛：

Oracle plans to deprecate the Java browser plugin in JDK 9. This technology will be removed from the Oracle JDK and JRE in a future Java SE release.

那該怎麼做？改用 Java Web Start 吧：

With modern browser vendors working to restrict and reduce plugin support in their products, developers of applications that rely on the Java browser plugin need to consider alternative options such as migrating from Java Applets (which rely on a browser plugin) to the plugin-free Java Web Start technology.

沒了 Java Applet，一般使用者就沒有理由安裝 JRE。普及率如果低到不像樣，Java 已死就會發生。但這是我們 Java 程式設計人員必須咬緊牙根度過的重要關頭，我們該思考如何用 JavaFX 設計出讓使用者方便操作的 RIA？如何用 Java Web Start 發行 Java Application？以此為基礎，吸引使用者安裝 JRE 來執行我們所寫的 Java 程式，而不是讓綁手綁腳的 Java Applet 使我們淪為笑柄。

只是，在 Android 盛行的時代，大家都用 Java 的寫法為智慧型手機設計應用程式，早就沒人用 Java 為個人電腦寫應用程式。就算 Java 軀體已死，做為靈魂的我們還是活著～
