不要總是站在「軟體開發」的角度學習「程式設計」

對「物件導向開發手法」來說，「物件導向程式設計」只是它其中一個環節，並且這環節被迫扭曲自己的「學術思維」來迎合那「經驗法則」，導致從「軟體開發」的角度來看待「物件導向程式設計」時，往往找不到物件導向程式設計的精神究竟何在～

所以，要學習正統的物件導向程式設計、或者認清初衷的物件導向程式語言，卻始終不得其門而入時，請暫且拋下你所接觸到的各家物件導向學說，轉而動手將傳統結構化程式語言的設計，改寫為物件導向程式語言的語法，直接以物件導向程式語言本身，去體會物件導向程式設計有了哪些改變？

這些改變才是物件導向的精神，物件導向開發手法只是進一步訴諸「這樣的精神有多神」。所以就像聽人說某家小吃攤東西多好吃多美味一樣，你光從物件導向開發手法訴諸的理念，是無法見證物件導向程式設計精神的，自己去吃、去體驗看看，才能了解為何物件導向開發手法會這樣做。

但物件導向開發手法有個通病是：「我都已經告訴你那東西是怎樣的味道了，你為何還跑去吃？」所以不要覺得熟於物件導向的人有多崇高，其實有他沒見識的一面。他以為他認知的是從本到末：「自己已經掌握的法則就是物件導向的一切。」但其實只是從本開出來的分支，不能涵蓋物件導向的一切～

也不要忽略「軟體開發」的立場學習「程式設計」

但也不要偏廢軟體開發的手法，畢竟 Java 就有許多用來滿足大型開發的語法，程式設計就是用來開發軟體或系統的；尤其靠笨重 VM 執行程式的 Java，它絕對不是用來寫指令功能的程式語言。

我們只是不要過度投機取巧突破 Java 語言的限制去建立開發手法，來彌補語法機制上的不足，甚至以為每跨越一次瓶頸就更邁向成功的一步。有時候就是得認命 Java 只適合那樣做，然後改變自己開發立場的需要去迎合 Java：「甚至放棄自己的需要。」否則，就是這些巧妙的手法，反而造成管理上的問題：「你變成不是在管理專案而已，還要管理 Java 是否繼續支援那些非標準做法。」

Java 語言無法滿足我們專案管理的需求時，那就調整我們的設計，看看能不能不要有這個需求，而不是建立新的手法去滿足這個需求。
