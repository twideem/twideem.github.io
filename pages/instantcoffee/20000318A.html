程式語言的演進

機械語言

其實機械語言並不是程式語言，而是最早期用來操作電算機的指令。

機械語言由 0 和 1 組成，因此你可以想像成，這是一種扳動整排「開關」，然後讓電算機工作的操作方式。每排可能有八個開關給你扳動，扳動不同開關組合會有不一樣的動作。而一台電算機可能有三排開關，例如第一排是機器動作、第二排是輸入資料、第三排是輸出腳位～

組合語言

由於 0 和 1 組成的操作指令不容易記憶，尤其每家 CPU 的開關情況不一樣，更加深機械語言無法做為共通語言來使用的障礙。

於是開始將常用的開關動作，標上英文字母做為符號，例如：JMP（跳進執行點）、MOV（搬移資料），這就是組合語言。

有了組合語言，各家 CPU 的操作指令，獲得更一致性的規範，讓電算機的使用更普遍。

由於組合語言是將電算機內部的開關動作定義個英文代號，因此使用組合語言必須熟悉電算機內部的運作原理，才能知道組合語言該做哪些事，這是一般人認為它困難的地方。[1]

但這不表示組合語言很難懂，其實組合語言比近代程式語言容易理解，例如在編寫程式的思維大致是這樣：「從某個腳位輸入資料，然後指定某個元件對這資料做運算，再輸出到某個腳位。」反而近代程式語言需要具備的思維更繁雜。

組合語言只是很難用而已，因為現代的應用軟體，是依賴作業系統在執行，要以直接控制電算機內部動作的組合語言來開發應用軟體會是費時耗力的巨大工程，開發應用軟體還是應該使用現代的程式語言才會有效率。

高階語言

因為組合語言都是一堆下達電算機工作用的命令，不夠口語化，因此發展出接近人類說話習慣的程式語言：「高階語言」。口語化最大優點，就是可以站在人類的想法去安排程式，不用再配合電算機的動作使人跟著一起呆板。

高階語言另一大特色，就是需要透過「編譯器」轉換為電算機看得懂的語言。至於電算機看不懂、反正人類看得懂比較重要的，則稱為「原始碼」。

既然以「高階語言」自居，之前的程式語言就被冠上「低階語言」的汙名，且這相對性的狀況至今依然延續著，例如 Java 比低階的 C 高階，後來 Ruby 又比低階的 Java 高階。

最早的高階語言中，以 FORTRAN 最有名，專門用於科學的工程運算。

結構化程式語言

其實高階語言只是在指令上接近人類的自然語言，語法使用上卻依然習慣低階語言的方便，結果寫出來的程式往往像組合語言一樣瑣碎，且自然表達的關係，每個人的思路多少有差異，除錯時往往找不到脈絡可循，可讀性反而比思路單純的組合語言糟糕：「看得懂在寫什麼，卻搞不懂在做什麼。」[2]

對此 Corrado Böhm 和 Giuseppe Jacopini 研究出「結構化程式設計」的寫法：

循序結構 - 程式流程由上往下執行，不要由下往上跳躍。
條件結構 - 程式流程可依據條件改變執行順序。
重覆結構 - 程式流程可重複執行。
為宣導結構化程式設計，Niklaus Wirth 發明 Pascal[3] 語言，並提倡用「流程圖（Flowchart）」讓結構化程式可以透過圖形來表現與檢視，對程式設計帶來極高助益。原本是在學院教授程式設計時使用，但後來其它高階語言皆參考 Pascal 的做法，實作出結構化程式設計的理念，可謂影響深遠。

另外，結構化程式語言這個時期的演變值得關注！因為可以這麼說，其實沒有「結構化程式語言」，它只是規定了「結構化程式設計」這樣做法的「高階語言」，從程式語言的演進來講，它只是高階語言的一種分歧。因此一般認為程式語言的演進到了高階語言就結束了，接著期待「第四代語言」。

然而，這個時期也表示「程式語言」其實演進為「程式設計」，語言（language）本身已經不是重點，程式開發手法（programming）才是議題。這種現象是值得關注的，代表從這時期開始，程式設計語言的進步，往往不在語言本身，只是在現有基礎上添加需求。了解這趨勢，學習上就不會感到困惑。

物件導向程式語言

隨著結構化程式設計成為規範，大家開始著重結構鞏固的效益。但實際進行結構化程式設計時，發現到有個破壞原則的結構存在！那就是在使用「函式」做為模組化的手法時，「變數」的生命期無法保存於函式。

有存在價值的變數，必須隨時提供資料的存取。而變數若屬於函式，函式執行完則生命期結束，消逝的變數無從被存取，於是要持續生命期的變數只能擺在循序結構（結構化程式設計的第一種結構），導致「並列流程」的函式往往被迫經由「序列流程」的結構存取變數，整個執行流程因此糾纏起來而複雜化。

堅持結構化程式設計的人，在序列的程式結構將遇上並列所帶來阻礙的問題，解決辦法就是讓程式的執行流程通通改為並列。這時發現，物件導向程式語言的某個特性，能夠滿足這方面的需求：

封裝 - 將邏輯與資料結合為單一個體。
但其實物件導向並不是為了解決結構化程式設計而存在，它甚至早在結構化程式設計以前就有了雛形。物件導向主要是為了程式的再利用，以提供程式靈活的擴充性而設計，其實物件導向真正的精神是：

抽象 - 將程式功能概念化。[4]
繼承 - 讓概念可以被描述。
多型 - 被描述出來的概念可以套用。
這也就是為何結構化程式設計的人，難以了解物件導向程式設計的主因，一開始就誤解物件導向的應用價值。

專案管理語言

物件導向程式語言普及後，人們首先感受的是，物件導向對專案管理有很大的效益，於是開始導入物件導向的專案管理手法。

然而，並沒有單一套程式語言叫做專案管理語言，而是綜合各種手法來實現。

最主要的手法，是將「物件導向」導入「軟體工程」的「客戶需求分析」「產品規格設計」兩部分，叫做「物件導向分析」「物件導向設計」。由於是種「手法」，它靠實務經驗的累積而來，無法從理論上學習，因此物件導向成為難以入門的專業領域。話雖如此，複雜的只是「物件導向專案管理」，如果你不逼自己非得學會這項手法，「物件導向程式設計」本身其實很單純。

但我們還是可以從 Iterative software development 的 Unified Process，獲得相當於理論的實踐方法，通盤學習到物件導向專案管理的學術專業。然而它是一套「宛如設備般笨重」的工具軟體，亦即沒有這套軟體根本沒用，因此大家流行採用的開發流程是 Agile software development 的 Extreme Programming，它純粹只是提供原則，你可以任意整合手邊的物件導向資源去實現，但也因此讓物件導向專案管理成為一種手法。

無論你選擇 Unified Process 還是 Extreme Programming，都需要一個能夠橫跨「分析」「設計」「編程」的共通語言，這時 UML 是最佳的選擇，它提供像是設計圖的畫法，能讓你在平面上建構整個專案與軟體的模型，做為開發過程中用來檢視的半成品，好比建築物與設計圖那樣。因此我們可以預期，若 UML 進一步被打造為可以執行的程式語言，那 UML 可以稱得上是「專案管理語言」。

另外，專案管理手法的使用下，「物件導向程式設計」只是遵照「物件導向分析」「物件導向設計」的規定行事，程式語言本身失去思考的權利，只是受專案的意志擺佈。這讓程式語言不再有創作上的藝術魅力，於是浮現「特定領域語言」，針對不同開發用途，制定一套專用程式語言，以試圖讓程式語言繼續在擁有自己的意志下開發案例。

最後，因為軟體工程的需要，物件導向被導入專案管理的領域而變得博大精深，但不表示物件導向非得如此不可。如果你開發的案例只像是為電算機安排行程，而不是浩大的工程必須縝密應付每一次變動，你不必為了掌握不了物件導向的專案管理手法，就認為自己不懂物件導向。這時回頭檢視「物件導向程式設計」原本單純的理念，你或許反而能找出真正適合自己的物件導向開發手法，那才是程式語言本身該有的創作性、藝術性。

終端元組件（俗稱「圖件」）

若雲端作業系統實現，原本「使用者藉由程式安裝應用軟體」的做法，變成「客戶端佈置一個操作介面，然後自行決定這個操作介面上，要整合哪些向伺服端訂閱的網路服務。」的做法時，用來進一步定義與描述的語言，就叫做網路終端語言。

之所以做法會變成這樣，是整個網路就像你的主記憶體那樣流動著作業系統所要的資料，就等你的作業系統來存取，所以更像雲端作業系統該做的事。

詭譎的是，這種做法並非真的打造一個雲端作業系統，而是制定一個「作業系統訊息」的「網路資料傳遞訊號」，才真正定位了「雲端作業系統」的概念。雖然畫工程圖時，畫朵雲的是網路的終點（伺服器），但其實網路整片的流動線，才是雲端的意義所在。

當然，目前看到的，都是「網路的終點」的雲端作業系統：「把東西放那上面而已。」所以網路終端語言依然是個空想。

制定「作業系統訊息」的「網路資料傳遞訊號」時，將賦予物件導向的抽象、繼承、多型機制，隨後便能經由通訊的架構，發展出「物件導向作業系統」。這讓物件導向程式語言長期以來的疑難獲得一次性解決，使得網路終端語言就另一層意義來講，其實是「物件語言」，亦即能夠不使用指令與邏輯來開發程式，直接就有完整的物件滿足你開發的需求。因為在本身就是物件導向環境的作業系統底下，不再有物件相容性的問題，物件的擴充性、再利用性都有了完美的機制。

然而，這個想法之所以成為主流，並不是技術本身受到關注，而是應用技術時意外獲得成功。為了廣泛應用「網路即作業系統訊息」的雲端作業系統，而制定的一套「視覺化元件」，成功取代「傳統程式指令」，成為「軟體設計」更有效的做法，因此名為「終端元組件」。也就是說，大家感興趣的是「這堆圖件如何架構雲端應用軟體」，然後上課或看書要學習這堆圖件的時候，第一章才會介紹「（網路傳遞的是作業系統訊息的）雲端作業系統」有哪些優點。

「圖件」取代「指令」後的一個特殊現象是，因為「服務」本身就能「最佳化」使用者需要訂閱的內容，因此「判斷」與「迴圈」居然不見了，這是現在的程式設計師難以想像的情景。但到時候的程式設計師也難以想像說，過去的人怎會想要用「條件式」去最佳化合適的服務內容，因為他們以為網路內容就是資訊軟體的本身與全貌。

溝通式智能

外表

聲控→溝通→學習

藉聲控與能夠自我學習的全智能系統溝通，來交換資訊。

內涵

服務→型別→陣列

將網路服務視為資料型別，成功加以陣列化，予以陣列式操作與運算，以這種高效率的運算速度，所誘發出的全智能系統。

影響

生技→寄生→愚民

為提升國民素質，各國開始以生物科技將溝通式智能裝置寄殖於人類身上，讓人民隨時隨地都能獲取網路服務的資訊來源。但後來開始出現總統高官與權貴才得以裝置最尖端產品的事情後，漸漸演變為人民被迫裝置後註定要落後於權貴的現實，而淪為一種政府控制人民思想的愚民化措施。
