演算法與資料結構

資料結構

一般來講，電腦程式設計常見的資料結構有…

線性結構　　　陣列（Array）
　　　　　　　鏈結串列（Linked list）
　　　　　　　堆疊（Stack）
　　　　　　　佇列（Queue）
非線性結構　　樹（Tree）
　　　　　　　圖（Graph）
行為結構　　　雜湊（Hash）

陣列

在記憶體的連續空間裡，存放一串資料，然後只要照著順序存取記憶體裡面的資料即可，因此是存取效率最高的資料結構！不過，要在資料中插入或刪除資料的話，必須搬動後面的資料，資料量大時，會有效率低落的問題。

這個資料結構正式名稱其實是「循序串列（Sequential list）」，但程式語言提供的「Array（陣列）」語法就是現成的「循序串列」，用久了便直接叫它「陣列」～

鏈結串列

如果資料筆數太長，就不見得能在記憶體裡找到足夠的連續空間。於是除了保存資料內容以外，還保存指向其它資料的記憶體位置，這樣就可將支離破碎的資料連接起來使用。

當然，必須額外透過指標來存取不同筆的資料，效率會降低。不過，中途插入或刪除資料的話，只要改變指向的記憶體位置即可，效率反而比陣列來得高！

只指向下一筆資料記憶體位置的，叫做單向鏈結串列，還指向上一筆資料記憶體位置的，叫做雙向鏈結串列。

堆疊

只從線性結構的頂部存取資料！不妨想像成堆高積木一樣，存入資料時不斷往上堆高，要取出資料時也是從最高位置著手，好像害怕從底部抽掉積木會垮掉一樣。這種結構的資料有「後進先出（Last in, first out. LIFO.）」的特性，存入資料的動作叫 push，取出資料的動作叫 pop。

雖然沒有強制規定，但適合使用陣列來設計堆疊。

佇列

從線性結構的尾部存入資料，再從頭部取出資料，也就是從線性結構的兩端來存取資料。這種結構的資料有「先進先出（First in, first out. FIFO.）」的特性，存入資料叫 enqueue 或 unshift，取出資料叫 dequeue 或 shift，或者跟堆疊一樣用 push 和 pop。

雖然沒有強制規定，但通常使用鏈結串列來設計佇列。

樹

以「節點（node）」的觀念來表示資料。通常由一個根節點開始，展開一或多個子節點，每個子節點又有一或多個子節點，枝枝繁生開來，整個資料結構就像一棵樹的形狀一樣。

善用節點的特性，可以讓你歸類資料，例如將大於 5 的資料放在某節點的右邊、小於 5 的就放在左邊，是一種能藉由資料的歸屬，來表現資料屬性的結構！這讓我們在設計演算法時能做更多元的思維，當解決問題遇到瓶頸時，不妨檢索樹狀資料結構的特性，說不定就能獲得靈感進而突破。

最基本的樹，是每個節點只有左、右兩個子節點的樹，叫做「二元樹（Binary tree）」，不但能用陣列做為樹的資料結構，然後套上數學公式（以 2 為因子的各種算式）即可找出節點，而且所有樹都能轉為二元樹，可說是樹資料結構的最基本型態。

雖然直接以陣列就能設計二元樹的資料結構，但裡面會留下一堆沒用到的空資料，深度越大，白白浪費掉的占用空間就越加驚人，所以若使用物件導向程式語言的話，有個最理想的做法，就是設計一個支援左右節點的類別，用鏈結串列來保存該物件！

還有一種高竿的人很喜歡用來提升資料進出與搜尋效率的樹資料結構，叫做「堆積樹（Heap tree）」，當你不滿於所有基本資料結構，想在其中有所突破時，就是上網查詢相關資料看看的時機了！你會發現問題不在於資料結構本身過於基本、簡陋，只要懂得換個角度靈活思考、應用，也能找出新的解法，醍醐灌頂後功力大增。

圖

像樹狀結構，但重點不是放在「節點」與其歸屬關係，而是放在連接節點的「邊（edge）」與其路徑關係來解決問題。由於節點與節點之間可以任意關聯，連接起來就像畫星座一樣出現各式圖形而得名。

數學早已有「圖論」的分支，所以使用圖形資料結構，可以幫助我們進行更具深度的演算法設計，解決更多難題。

A 點和 B 點之間，有 A 到 B 和 B 到 A 兩種向性，可見邊是有單向和雙向之分的！因此通常使用二維陣列來設計圖的資料結構，將 A 視為 1、B 視為 2 的話，那要表達 A 點到 B 點有邊，就將 [1][2] 填上資料，B 點到 A 點有邊，就將 [2][1] 填上資料。這資料可以填上 1 表示 True，也可以填上長度，要計算最短或最長距離時，顯然這個資料結構本身就是模型，有利於找出解法！

雜湊

可視為一種用關鍵字來儲存資料的結構，因此在取出資料時具有最快的效率，例如給予關鍵字直接就能取得資料，是設計「搜尋演算法」時非常受歡迎的資料結構。

但是在存入資料時，可能因為用關鍵字來保存資料的手法設計得很複雜，而影響效率。

不過，也因為保存資料的手法可能是透過某種數學計算技巧改良而來，所以有時候設計出來變成一種可以節省空間的資料結構，而被當作能夠壓縮資料的結構亂用。

估算程式

設計演算法時，需要懂得評估自己程式的執行效率。本節示範如何估算程式，以及藉由估算結果提升程式執行效率。

N4

所謂估算程式，就是在整個程式裡，找最耗時的區塊來做為基準，然後降低其執行次數。

來看底下的範例，它從 a 裡面抽出數字四次，相加起來能夠等於 e 的話輸出 true，否則輸出 false。


true
首先來看 boolean i=false; 這行程式，它比起下來的 for 迴圈來說，影響的時間微不足道，所以要評估程式的演算法時，沒必要理會它，我們需要改良的對象只有耗時間的 for 迴圈！

而這個程式來說，耗時間的 for 總共用了四次，而且每次都與前一次的執行次數一致，於是我們評估這程式的演算法步驟為 N4。

也就是以迴圈這耗時間的程式為基準：N，然後這程式等於四道迴圈：N x N x N x N，因此簡稱為 N4。

這個 N4 程式，在我一台 2.8GHz 單核心處理器上跑，大概要花兩分鐘的時間。

N3 log N

底下程式將最內部迴圈的 a[m]+a[n]+a[o]+a[p]==e，改用 java.util.Arrays 內建的 Binary search，搜尋這則 e-a[m]-a[n]-a[o] 式子的值：


true
這樣可以消除一層迴圈，因此外層迴圈剩下 N x N x N 估算為 N3，而最內層的計算時間只剩原本的一半不到，這種情況叫做對數時間，估算時寫作 log N。所以整個估算結果表示為 N3 log N。

以範例的資料長度 a.length 是 300 來講，N4 與 N3 log N 的差距為：

3004 : 3003 log 300

約 120 倍，現在這 N3 log N 的程式只剩十秒鐘不到就能執行完畢。

N2 log N

十秒還是很久！不曉得哪位精於設計演算法的人，發明了將陣列分成兩組，結果再消除掉一層迴圈的做法，真厲害[3]：

現在迴圈剩下 N x N 估算為 N2，而最內層的計算時間依然是 log N 的 Binary search，所以整個估算結果表示為 N2 log N。

這個 N2 log N 演算法的程式，能在三秒鐘之內完成。

演算法的分析

在需要更精確評估演算法的場合，會用到以下符號：

Ο(ƒ)　實際執行次數 ƒ 大於等於評估次數。
Ω(ƒ)　實際執行次數 ƒ 小於等於評估次數。
Θ(ƒ)　實際執行次數 ƒ 等於評估次數。
有時候還會用到這個符號：

ο(ƒ)　實際執行次數只會大於評估次數，也就是不會等於評估次數。
評估演算法的效率必須講究實際，因此都是指「最壞的情況下程式執行次數為多少」，所以預設都是以 Ο(ƒ) 做為基準。以上述範例的 N4、N3 log N、以及 N2 log N 來講，更精準的寫法是：

Ο(N4)
Ο(N3 log N)
Ο(N2 log N)
當我們估算出程式的演算法步驟後，就要朝降低次數的方向邁進。這時我們會參考如下稱之為「成長率」的排列：

Ο(С)
Ο(log N)
Ο(log2 N)
Ο(N)
Ο(N log N)
Ο(N2)
Ο(N3)
Ο(2N)
資料量（N）大時，越上面代表越佳時效的演算法，最下面是最糟糕的情況。藉由估算，可以找出程式執行效率不佳的問題在哪，然後參考成長率，來改良程式設計。不過，資料量小的話，設計或套用演算法反而可能變慢，不妨自己將 N 套入上述排列的結果看看，雖然 N=1000 時落差很大，但 N=2 時卻多此一舉。[4]

另外，演算法的評估可是一門學問，甚至有些演算法該如何評估其成長率會引起一番在爭論。例如 Shell sort 這個演算法，在不同資料結構會有不同的表現，最後是經由反覆實驗才被估算為平均 Ο(N7/6) 的非典型[5] 案例，可見演算法分析有時候不是那麼簡單的事。

時間與空間

以上介紹的都是「時間複雜度」的演算法。

然而，如果發明了時間很快的演算法，但必須用 1TB 的 RAM 才能運作，那恐怕也稱不上好的演算法[6]～

因此有時候「空間複雜度」也是演算法設計的焦點。

排序演算法

常見的有四大類，以及各自的改良版本：

交換排序（Exchange sort）　 快速排序（Quicksort）
插入排序（Insertion sort）　希爾排序（Shellsort）
選擇排序（Selection sort）　堆積排序（Heapsort）
合併排序（Merge sort）　　　串列排序（Strand sort）
Java API 的 Array 與 List 已經有專家們設計好的排序，一般而言我們不可能寫出比他們更快的排序演算法，所以寫 Java 程式很少會需要自己設計排序。

但身為程式設計師，能夠寫出應急用的簡單排序演算法是基本能力，至少要挑一個來熟記～

Exchange sort

交換相鄰兩資料，把小的逐步往前放：


1234
資料排序的演變過程為：4321→4312→4132→1432→1423→1243→1234。由於資料就像氣泡往上冒一樣，所以又叫「氣泡排序法」。

不過把大的資料逐步往後放，程式碼會比較好背：


1234
資料排序的演變過程為：4321→3421→3241→3214→2314→2134→1234，可以戲稱為「沉石排序法」。為了提升效率，可將內層迴圈的條件句改為 n<array.length-1-m 逐步縮小比對的範圍。但這是應急用的，好背就好，省略無所謂。使用交換排序法，目的是會動就好，沒在講效率的～

Index Sort（索引值排序）

以上三種排序演算法，太久沒用還是會忘記，臨時才想到要用，往往測試到覺得沒問題時，已經耗掉十分鐘～

如果要趕在一分鐘內搞定一個排序程式來用，那就用另外一個陣列來保存資料，利用陣列索引值本身就是一種順序的特性來設計吧！

尤其以上三種排序演算法出現速度很慢的情況時，這個以空間成本換取時間效率的排序方式，更是解決問題的好招。

這種演算法的缺點，就是資料很大的話，會浪費很多記憶體空間。再怎麼以空間成本換取時間效率，看到 long[] sort=new long[4294967295]; 這行程式還是會心揪一下！

這時應該選個更適合的資料結構取代陣列，例如鏈結串列，來節省空間～
