Relationships (類別之間的關係)

其實這些線條，才是有益於檢視物件導向的重點！

Association (關聯性；結合)

A{ B }
類別 A 用了類別 B；可能做為屬性，也可能是操作時傳遞的訊息。

Dependency (相依性；依存)

A{ B }
類別 A 不只用了類別 B，而且會受其變動所影響。

Generalization (一般化；泛化)

A extends B{}
類別 A 繼承類別 B。

Realization (實現)

A implements B{}
類別 A 實作介面 B。

Aggregation (聚合)

A{ Vector<B> }
或

class A{
　private B b;
　public A(B b){
　　this.b = b;
　}
}
這是目前 UML 少數交代不清、有待新版本賦予更清晰語意的表示法！現在的主要意思是：類別 A 有一部份是類別 B[1]。

Composition (組合)

A{ Composition<B> }
或

class A{
　private B b;
　public A(){
　　this.b = new B();
　}
}
比「聚合」有更嚴謹的定義：類別 B 一次只能被一個類別使用，且類別 A 銷毀時類別 B 跟著銷毀。

由於現代新興的物件導向程式語言，都有自動銷毀物件的機制，因此顯得不適用，常常主要意思被改為「類別 A 由類別 B 組成」來用[2]。

[1] 應該說「類別 B 是類別 A 的一部份」，但我們讀 UML 的類別圖時，習慣從線尾指向箭頭，所以說得這麼彆扭。
[2] 也就是說，對 C++ 程式設計人員來說，看到 Composition 的圖形表示法時，會覺得要注意物件銷毀的問題，但是對 Java 程式設計人員來說，可能會覺得要注意的是物件生成的手法。同樣一個圖形表示法，卻可能會有截然不同的意圖，這表示根本不夠成熟到應該放入 UML 裡面！因此建議不要使用它，甚至連「先做好團隊溝通，定義好 Aggregation 與 Composition 的語意，再使用這個圖形」都不要，因為即使當作萬不得已時才使用的特例表示法：「你團隊為何不自己特例一個更棒的圖形來表示？」用這官方規定的圖形表示法，比你特例的圖形表示法，更容易誤導人「各說各話」。
