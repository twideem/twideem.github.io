如何使用本單元下載的範例檔

檔案結構

每個下載的範例，裡面的檔案資料如下：

[binaries]編譯好的程式會放在這個資料夾。
[sources]存放原始碼的資料夾。
build方便 Linux 使用者編譯與執行的 Shell Script。
build.bat方便 Windows 使用者編譯與執行的批次檔。

如何編譯與執行

雖然你可以自己輸入 javac 和 java 指令來編譯與執行範例程式，但使用範例提供的 build 更為方便。

Windows

直接滑鼠雙擊 build.bat，即可進行「編譯」與「執行」的動作。

批次檔跑完，會停留下來等待按任意鍵結束，避免錯過指令模式顯示的訊息。

Linux

請開啟終端機，切換到範例所在目錄位置，輸入 ./build 或 sh build 來編譯與執行。

如果是 Lubuntu 的話，用檔案總管開啟範例的目錄，然後按 F4 會自動以範例的目錄位置，啟動終端機，相當方便。

補充說明

１

每個範例使用的批次檔與 Shell script 都是一樣的，不會為了個別範例的特殊條件，來修改指令參數與組態設定。

因此當你練習範例遇到錯誤時，可以排除是不是這個範例設定了不同的編譯與執行方式才發生問題，專心把焦點放在原始碼與套件身上。

為了做到這點，程式需要 import 第三方 package 時，本站提供的範例採取先解開 JAR 檔案然後丟到 binaries 資料夾的做法，也就是與範例編譯出來的程式放在一起，這樣就不用設定 classpath 也能 import 第三方的 package。

但這只是為了範例方便性而折衷的做法，正式場合不建議以這種方式使用第三方套件，否則你的專案可能會有成千上萬個 *.class，要封裝暨部署時會很頭痛[1]～

２

因為有可能使用新的 JDK 編譯範例，卻用舊的 JRE 執行程式，導致練習範例時發生問題[2]，所以本範例寫在 build.bat 和 build 裡的 javac 和 java 指令，是透過 JAVA_HOME 來叫用。

在使用 build.bat 或 build 執行範例編譯出來的程式發生問題時，請檢查你 JAVA_HOME 的路徑，結尾是不是用了 \ 或 / 符號？例如：「C:\Program Files\Java\jdk\」或「/usr/java/jdk/」。

請把最後的 \ 和 / 移除，才是正確的 JAVA_HOME 路徑。

３

為避免未來新版 JDK 的編譯條件變嚴苛，導致範例在將來可能無法通過編譯時，讓初學者誤以為是原始碼的問題，所以本站範例都已經先編譯過，證明原始碼曾經通過編譯，沒有問題。

４

如果需要讀寫檔案，則一律放在每個範例的根目錄，避免翻找檔案放在哪個資料夾。

５

本站預設初學者使用的作業系統是 Windows，因此原始碼使用 CR+NL 的換行格式，這樣使用「記事本」也能檢視與編輯。

但使用的並不是 Windows 含 BOM 的 UTF-8 字元編碼，因為有些場合無法辨識 BOM 記號，結果造成非預期的執行狀況，徒增困擾。

６

本站預設教導的對象是新手。而在我教導程式設計的經驗裡，發現剛開始學習程設計的人，希望直接能看到每項議題的新功能！因為對新手來說，範例中夾雜太多可有可無的程式時，容易被程式結構的繁雜給混淆，導致搞不清楚重點在哪，降低學習時效。

因此本站的教學範例，風格上避開了「先宣告參照再來使用物件」的慣例，盡量直接建立物件來凸顯主要功能在哪[3]。遇到 try...catch... 時，也是整塊程式放在一起不細分，讓重點聚焦在一起。重複的程式碼也是如此，寧願複製貼上讓程式碼由上而下連貫起來閱讀，也避免設計成函式變成跳躍式判讀。

但這些都是不良的程式撰寫習慣！希望正式應用在開發場合時，使用者能修正編碼風格為…

．多宣告變數名稱提升可讀性，多定位參照降低建立物件的成本以提升執行效率。
．只讓需要拋出例外的程式存在 try 區塊，不需要的就隔開，並且更仔細 ctach 例外。
．善用函式整理重複的程式碼，善用繼承整理重複的函式，善用多型整理重複的繼承。
本站只是方便新手能夠直接看到想要的主題功能才那樣寫範例，實際應用時那些都是壞習慣。
