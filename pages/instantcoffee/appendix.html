我對 MVC 架構的看法

我覺得 MVC 只是站在「軟體設計」立場，來重新定義「硬體設計」的 PU + OU + IU 三者，使「運算、輸出、輸入」的概念，能更適切反映圖型介面軟體使用者內心的想法。

使用圖型化軟體的人，感官的認知是：

軟體功能（M）、操作介面（V）、使用方式（C）

而不是使用命令列時認為的：

計算（P）、輸出（O）、輸入（I）

為何要重新定義？因為兩者確實是有差異的。對軟體使用者來說，操作介面（V）往往等同於軟體功能（M）本身，但是對硬體使用者來說，輸入（I）感覺上才是計算（P）的本質。

換句話說，使用圖型化軟體時重心放在「輸出」，操作命令式硬體時重心放在「輸入」。

重新站在 MVC 的角度來作設計，能讓我們更精準地回饋使用者想要的事情。

沒有立場地勇於客觀剖析出優缺點

如果你從《程式語言的演進》一路讀到《不要違反 Java 的物件導向思維》，應該可以發現，雖然我常常指出 Java 有哪些缺點，但並不是為了反 Java 批評用的。

而網路上很多人有種刻板觀念，看到有人指出某甲缺點，就認為他反某甲，看到有人指出某乙優點，就認為他挺某乙。

做為工程師，你不能繼續秉持這種二分與對立的觀點來看待事情，必須勇敢認清自己所用的東西有那些缺點；認清事實，你會發現自己能夠客觀地應用這些優缺點，成功扮演工程師縝密洞悉問題的專業角色。

記住這句話：「優點、缺點，其實都是特點！」我們是基於剖析出「特點」的視野，指出了「優點」與「缺點」，而不是為了支持或反對而挑出有哪些優缺點。

除此之外，也要有勇敢指出缺點的素養，不要因為怕被人戴帽子、栽贓、攻訐，而違背良心附和他人說話。有句俗話說：「自古聖賢皆寂寞」，專業人士也是一樣，當你擔起專業人士的職責大任做出任何評估，就會有一群低俗的人說三道四，認為你是某派立場的人；專業的你必須忍受「客觀剖析優缺點各有其用途」時的寂寞[1]。

把握時間剖析專業問題，不要浪費時間向低俗的人多做解釋。

以《不要違反 Java 的物件導向思維》這篇文章來說，如果礙於低俗的人說三道四，我們就無法針對「Java 不是純正的物件導向程式語言」這件事，進一步探討出兩全其美的解決做法。他們會抨擊你不該批評 Java，認為你是反 Java、不想使用 Java 的人；但其實就是為了更深入地使用 Java，才會正視 Java 的缺陷在哪裡，來進一步盤算該怎樣處理。

然而，對只知道採二分與對立觀點來攻擊你的人，再怎樣解釋都是沒用的。況且，我們需要探究的重點是事件本身，而不是浪費時間來證明「我不是如你所說某派立場的人」，所以也不建議多做解釋。

會讓你的專案議題給擱置下來，扯到其他話題的人，請不要跟他認真。認真的話，問題不是出在對方身上，而是出在你自己跟著低俗。

StringBuilder 使用時機

嚴格來說，String 物件所保存的內容，並不是字串資料，而是字串資料的位址。所以每次變動字串資料時，它會新建一個用來存放新字串資料的物件，然後指向新物件的位址。

StringBuilder 則實際保存一份字串資料做為自己的內容，當你變動資料時，它直接修改自己所保存的那份資料，而不是配置新的資料。

所以，當你頻繁改動 String 物件所保存的字串內容，其實是頻繁改動 String 物件指向新的物件，這時使用 StringBuilder 比較適合。但如果物件所保存的字串幾乎是唯讀的用途，只讀取不改寫，那使用 String 比較省事。

re:SAX 基本範例：快速剖析 XML 文件裡的資料

大多數人視 SAX 為 Java 內建 XML 處理套件中最基本的解決方案，於是有「先會 SAX 再學 DOM」的傾向[1]。但我個人則視 SAX 為遇到 DOM 處理效率不彰時才使用的方案，可以的話我盡量不使用 SAX。因為它並沒有讓原始碼更 simple，反而增加維護的複雜度；SAX 所 simple 是資料本身。

我喜歡在 DOM 將 XML 模型轉為 SAX 來剖析，資料結構的處理情況會更直覺。

[1] 畢竟事件驅動的 SAX 可以剖析由程式碼直接描述所產生的 XML 資料，這點來講，DOM 只能剖析存在於記憶體的 XML 結構。但我從未遇過這樣的開發場合，因為 XML 不是這樣用的！
[2] SAX 只能讀取 XML 檔案的資料，不能將資料寫入 XML 檔案。

使用 Lambda 的時機

Java 是訴諸「物件導向程式設計」的靜態型別語言，亦即由編譯器規範我們的原始碼應該符合哪些守則，好讓原始碼能夠依據前後文的關係詮釋語意，以利大型專案的管理。

而 Java 的 Lambda expressions 像是訴諸「超級指令」的動態型別語言蜜糖，它用來讓「前後文簡化在一起」。對程序導向程式設計來說，因為減少跳著讀程式碼的工作，所以能夠提升可讀性。但對物件導向程式設計來說，雖然不用「跳讀」，卻失去前後文關係的推演，變成得在簡化後的程式碼「判讀」來解釋語義，而不是詮釋語意；注意這裡用的語義（semiotics）與語意（semantics）是不同字眼。

我個人對 Java 的 Lambda 實用度持保留態度，可能要再過一兩年時間，累積眾家將這語法機制導入大型專案的經驗，匯集到下一本《Effective Java》，才能知道怎樣用才好？甚至該不該用～

寫完即丟的程式碼用 Lambda expressions

一些寫完即丟、不需要重複使用的程式，就用 Lambda 來處理。不過這方面 Java 的 Lambda 沒有 JavaScript 的一級函式容易使用。

以 Stream API 實作介面功能

在「實作」部分用 Stream API 讓細節更簡短、清晰，「介面」部分繼續用過去物件導向的設計與實作分離。也就是只在「實作」物件公開的「介面」時，才用 Java 官方基於 Lambda expressions 設計的 API 讓內部細節更簡短而清晰地表現功能設計，至於實作與介面分離的做法並沒有變化，物件導向設計依然是物件導向設計。

不自己設計基於 Lambda 的功能來用

我傾向用 Java 現成的 API、或 Comparator 和 Runnable 這種「標準案例」就好，而不考慮自己設計基於 Lambda 的功能來用。因為這些經典用法已經像是設計模式一樣，大家熟悉它寓含的前後文關係了。如此既能提高程序導向式的可讀性，又能降低物件導向式的判讀性。

不要妄想用函數式風格寫 Java 程式

Java SE 8 的函式不是物件，所以不適合寫函數式風格的程式。加上 Java 是強型別（不會自動轉型）且完全物件導向的程式語言（傳入的參數要寫得落落長），寫出來的函數式程式也不好看。

如果你希望用函數式風格寫程式，請改用致力於將物件導向程式設計與函數式程式設計完美整合在一起的 Scala，因為 Java SE 8 用 functional interface 取代 first class function 的設計，顯然在抗拒函數式程式設計。

用匿名類別直接跑程式碼比較方便

既然函式不是物件，Java 的匿名函式也就無法直接寫個 class 來調用 method，做這件事要繼續用匿名類別：


習慣在 JavaScript 將一級函式物件做為參數與傳回值來跑程式碼的人，在 Java 的 Lambda expressions 容易發生鬼擋牆，因為要用匿名類別來做匿名函式該做的事就跟見鬼了一樣。

特殊體驗

在使用 Java SE 8 的 Lambda expressions 時，我發現自己會有兩種目的，久而久之，分開稱為「匿名函式（Lambda）」和「匿名算式（Lambda expressions）」。

也就是有時候我是為了「傳匿名函式作參數」，這時我稱自己正在 Lambda。有時候是為了「寫匿名的算式」，這時我稱自己在用 Lambda expressions。

不過這樣只會更難認知 Java SE 8 的 Lambda expressions 是怎麼一回事，所以看看就好，不用真的這樣區分。我只是把自己的特殊體驗 PO 出來分享而已～

[1] 或者說不是很認同它的原理與理念，所以不完全接受官方說法，以免扭曲我對 Lambda expressions 的認知。我居然還要站在 Java 為了相容語法舊包袱的立場，才能正確使用 Lambda expressions，然後還得定義說這樣的 Lambda 才是對的…變成這樣不太好，所以我不是很清楚 Java SE 8 設計 Lambda 的原理和理念。只是，試著去了解時，對 Java 維護團隊能夠一邊相容語法舊包袱、一邊實現新功能進來的智慧，感到讚嘆不已～

re:使用 Lambda 的時機

Java SE 9 推出半年後，Java SE 10 都推出了！Java SE 8 的 Lambda expressions 用了這麼久，我的看法也變明確：「函式介面的 Lambda expressions 用起來規則太複雜，讀起來又帶有不確定性，沒有一級函式的直覺與實用性。所以在 Java 寫物件導向程式時我不用 Lambda expressions，只有在操作的 API 是函數式風格時才用。」

除了操作 Stream API 以外，我不使用 Lambda expressions。

可以跑 Lambda 的時機、以及類型推導的規則，都太複雜了，不是每個寫 Java 的人都能熟記與掌握。所以大量用了 Lambda expressions 的 Java 程式，只會造成程式設計師的壓力。寫 Lambda expressions 看似字數減少更簡潔，其實要讀懂在做什麼事，得用推測的，不像過去光看物件的操作方式，便清楚掌握會做哪些事。於是我決定不使用 Lambda expressions，以免閱讀程式時帶來不確定性。

我只有在 Stream API 的場合才會使用 Lambda expressions，因為把範圍限縮在這套 API 的話，是很容易熟記與掌握的！且這套 API 函數式設計得很好，行為表達很明確。但在其他寫 Java 物件導向程式設計的場合，我是不用 Lambda expressions 的。

出現在 Stream API 的 Lambda expressions 不難理解在做什麼，但出現在其它 Java 程式碼的 Lambda expressions，有時候要推測才知道做什麼，所以閱讀原始碼時若要修改程式，理解效率會下降。不過，閱讀原始碼時若要追加程式的話，因為容易找出重點在哪個環節，效率是上升的。但就我個人而言，易理解比較重要，所以我不會逮到機會就用 Lambda expressions，而是沒事別用。

空前設計 v. 先前就有的一級函式物件設計

當初我對 Java SE 8 拿 interface 來實現 Lambda expressions 感到佩服，這是集眾菁英們聰明才智完成的空前設計！所以我安裝 Java SE 8 後，把玩的不是 Lambda expressions 好不好用，而是玩味 Lambda expressions 設計上的細節，讚嘆人類智能表現總是一再克服困難，感慨我這輩子都無法達成這些神人的成就。

實際用了以後，看來抄別人的做法，加入 function 型態做為 first class，讓隨處都能 Lambda expressions 才是較好的做法。因為充滿限制讓程式碼與思維都無法保持一致性，最後為了保持 Java 一致性的精神，我選擇無視 Lambda expressions 的存在。

Java 的 Lambda expressions 是展現人類高度智能表現的空前設計，但我寧可用先前就有的一級函式物件設計～

就像在 Python 我不想把程式寫成 class 來用，在 Java 我也是同樣心情，不想把程式寫成 Lambda expressions。Python 的類別寫法又醜又蠢，Java 的匿名函式也是一樣。

萬惡的 null

在「有還是沒有」雙重身份變來變去的 null，很容易形成系統設計上的瑕疵，但不見得就是萬惡之源。

例如有些資料庫軟體，很明確地使用 null 做為欄位的初始資料，然後軟體私底下自己會處理類似 java.util.Optional 的設計。在這樣的情況，我們發現使用 null 做為初始資料的設計，比為了防範 null 而導入 Optinoal 要來得優雅。

但如果我們無法明確定義 null 是什麼就使用它，然後有一天 null 又成為除錯的禍源時，那麼長期以來在「語法編譯」「執行例外」都百分百合乎規矩的它不但很難追蹤，甚至連「邏輯問題」中它也存在得很合情合理時，這隻 null 會像歷史懸案一樣難搞，不禁咒罵：「萬惡的 null～」

很多大師級人物都主張不要用 null，但雖然 null 有它禍害的一面，在某些場合也有它優雅的一面，因此我不主張「萬惡的 null」。我認為應該先試著定義 null 能否是很明確的單一什麼，能的話優先使用 null，不能的話就引進 java.util.Optional。

當然，幾乎都是無法明確定義出 null 是單一什麼的情況居多，久而久之就認為 null 什麼都不是，直接導入 Optional 來定義它，或者放任 null 成為萬惡的根源～

暫且不要導入模塊化

Java 9 最大的亮點，就是新的模組化方案！

舊的模組化有許多弊端，像是以類別路徑為基礎的封裝做法，新舊模組只會彼此覆蓋，互不相容，新的模組化方案有更多細節的微調，能解決長久以來的積弊。

然而，Java 9 的模塊化功能，是屬於平台上的特性，而不是語法上的特性，它對程式設計沒有多大幫助，不要想說這是新的特性就急著導入它，應該順著未來趨勢的發展，被動地慢慢導入它。

Java 9 雖然導入新的模組化方案，但連 Java 平台本身都還未通盤地以模塊化架構規範套件，可說是在準備階段而已。因此目前還不需要去關注模塊化的議題，等到 Java 平台自己也理出頭緒規範好模塊後，我們再「被迫」照規矩導入即可。這感覺就像你不見得非要把 Java 程式封裝成 JAR 來用，但等大家都用 JAR 釋出套件時，我們就必須了解怎麼將 JAR 套件導入到程式裡面。

我建議在 Java 9 的階段，暫且不要去管模塊化這功能，搶鮮導入模塊化不會帶來助益，反而會有徒增困擾的風險。等 Java 平台的模塊化架構好了，我們再來了解怎樣切割，讓程式依賴更小的環境便能執行。目前 Java 9 可以切割的，就 Java SE、JDK、JavaFX 這幾塊，雖然對客戶端應用程式的發布還是很有用，但還是不小塊，吸引力不大。

至於為何 Java 9 給了鋸子，自己卻沒鋸好東西來，只是示範這鋸子真的能鋸東西而已？我想，或許是為了在 Java 10 或 Java 11 有新的資料型態架構後，再全新模塊化 Java 平台吧？純屬臆測就是了～

Java Swing v. JavaFX

由於 JavaFX 比 Java Swing 容易使用[1]，因此網路上看到有人在推薦 JavaFX 時，會把 Java Swing 說成是很難的技術，好像只有專家才知道怎麼用。

這點我很不認同，比起 C++ 語言的 Qt、GTK+、wxWidgets…等視窗程式設計套件，不僅寫法繁贅，光部署套件好讓它正常運作，就容易讓人搞到頭大了，相較之下用 Java 的 Swing 來開發視窗應用程式，已經簡單、易用多了！使用 Java Swing 只需要關注元件有哪些功能可操作，完全不需要再管視窗程式是如何與作業系統交換訊息，因此你在 Java API 文件查到什麼功能，直接就能敲進程式碼跑看看情況怎樣，哪有什麼技術高深之處？（查 API 是很累沒錯，但沒什麼高深的技術吧？）

Java Swing 問題出在它提供的元件近八年來沒在擴充，已經不敷使用[2]，許多新版作業系統標榜的酷炫功能寫不出來；想自己根據 Java Swing 架構設計出來用，確實是專家才知道的事。這點 JavaFX 不僅有各式新穎的元件可用，而且因為能夠 RIA 的關係，可以很有彈性地用更多種架構自訂元件，還經常寫寫 CSS 就能改造出來。

Java Swing 只能說是功能有限，能做的事不夠多，但要說它很難、只有專家才會用，實在太離譜了：「Java Swing 其實連新手都會用。」（而且要求不多的話，其實很好用。）

當然，有了既能寫設計 Rich Internet Applications 又能打造 Desktop Application 的 JavaFX，只能 Desktop Application 而且很兩光的 Java Swing，被取代是遲早的事。不過，用 Java 寫 Application 一直很冷門，Java 熱門的是做為企業伺服端的解決方案，所以網路上不少人說 Java Swing 不會被淘汰，因為它從來就沒流行過 XDDD

JavaFX 讓 Java 更適合打造 Application，如果 Java 因此能成為開發應用軟體的熱門語言，網路上有各式各樣 Java 打造的 APP 可用，那真是我們 Java 程式設計師這輩子冀望見到的榮景啊～！也希望 Oracle 能翻譯官網那豐富的 JavaFX 教學文章，幫助大家不用買書，就能快速、有效掌握 JavaFX 的用法，好讓 Java Application 能推廣起來。

[1] 應該說 JavaFX 有更多靈活的方式可以使用，所以容易使用。否則 JavaFX 整套學起來，並沒有比 Java Swing 單純。
[2] 連選擇日期的對話視窗，盼幾年了就是等不到 Orz

舊版 JRE 跑新版 JDK 編譯出來的程式？

Linux 安裝 IcedTea 做為 JRE 時，它會以系統變數的路徑，取代環境變數的 JAVA_HOME 路徑。而 IcedTea 版本通常較舊，結果雖然用 JDK 的 javac 來編譯程式[1]，執行時卻因為系統變數優先，跑 IcedTea 的 java，而不是 JAVA_HOME 的 java。

Windows 也有可能。例如有些人透過修改 JAVA_HOME 環境變數，來切換新舊版 JDK 編譯程式。剛修改時一切正常，但下次開機時，Microsoft 為了加快開機速度所採用的混合啟動技術，有可能套用之前殘留的系統變數值。發生這種情況的話，會變成一下用未修改前的新版 JDK 路徑編譯程式，一下卻用舊版 JRE 來執行[2]。

[1] IcedTea 沒有 javac，因此不會取代 JDK 的 javac。
[2] 這時要用傳統的、開機很慢的方式關機，讓作業系統重新整理系統變數。
