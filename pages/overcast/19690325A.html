底線命名原則

variable：可正常存取的變數。
variable_：變數名稱重複時，習慣上在後面加個底線。
_variable：不建議有人存取這個變數時，習慣上在前面加個底線。
__variable：類別繼承時，不會共用該變數。

libui 圖形化介面

比起龐大到上百 MB、上百個檔案的 GUI 平台系統，有個只需一個檔案、大小不到 300KB 的 GUI，能跨平台、跨語言使用，叫做 libui。

但 Python 已經內建 tcl/tk 圖形化介面，而且也算輕巧、簡單，所以很少有人安裝 Python 時不願安裝 tcl/tk，應該用不上 libui。

不過還是蠻有意思的！本文將示範直接把 libui 丟進 Python 程式專案裡，在不安裝模組的情況下，任何安裝過最基本 Python 的人，都能執行你用 libui 寫出來的視窗應用程式～

下載

libui

首先下載 libui：

http://github.com/andlabs/libui/releases/

每個版號都有一個 Assets 可按，會跳出各種版本的下載清單。Windows 且 64 位元的話，下載 libui-alphaX.X-windows-amd64-shared.zip。

pylibui

然後下載 Python 的 libui 程式庫：

http://github.com/joaoventura/pylibui/

按 Clone or download → Download ZIP 整包捆回家。

設定

將 libui-alphaX.X-windows-amd64-shared.zip 裡面的 libui.dll 解出來，放在你的 Python 程式專案裡。

將 pylibui-master.zip 裡面的 pylibui 解出來，放在你的 Python 程式專案裡面。

修改 pylibui\libui\__init__.py，將 lib_dir = os.path.join(curr_path, 'sharedlibs') 改為 lib_dir = '.'。

這樣就能在不安裝模組的情況下，直接把 libui 打包在你的程式專案裡面，不管誰都能執行你的程式專案了。

Hello, world!

pylibui-master.zip 裡面的 examples 有各式寫好的範例，其中 window.py 是最簡單的空白視窗程式，不妨解出來執行看看，當作 hello world 吧！

執行不起來的話，不妨下載把上述都打包好的範例試試看：19690325A.7z。

教學

libui 是給沒內建 GUI 程式庫的語言用的，像是 Go 和 PHP 這些語言。只要程式語言支援 DLL，就能輕易使用 libui 寫圖形介面程式。Python 已經內建 tcl/tk，相比之下也沒什麼缺點，所以就不教怎麼用 libui 了～

如果真有需要把 GUI 打包在程式裡面，讓不想安裝 tcl/tk 的人也能執行視窗應用程式，再研究怎麼寫 libui 程式吧！

從濫用 XML 的時代記取教訓

2001~2010 年代 Java 熱門時期，能夠跨平台解析資料的 XML 被大量使用，然後變成一種流行，好像沒用到 XML 就沒走在技術前端一樣，於是寫 Java 程式的人，用 Framewrok 都被要求設定個 XML 組態檔。不同組態檔都有其結構要遵循，於是就有結構龐大複雜，好像把整套系統通通移植到 XML 身上來操作的組態結構，沒設定好就看不到結果。於是時間都花在設定 XML 組態檔，無法專心寫 Java 程式碼，那種情況就像每寫幾行 Java 程式碼，想要執行就要再設定好幾行 XML 組態，既麻煩又折騰。最後，除錯時發現問題經常出在 XML 組態檔，而不是 Java 程式碼，XML 就變得惹人厭了。

如果要用 XML 成為你程式的一部分，請讓 XML 成為你程式中可有可無的一部份，而不要成為非有不可的一部份。例如 CSS 對 HTML 就是可有可無的一部分，套用的話，可以呈現出你精心設計的版面，不套用的話，頂多呈現瀏覽器預設的樣式，而不是鬧脾氣死給你看。

對 XML 使用時機的建議是：「想將資料與程式切割開來時用 XML。」而濫用 XML 的時代，則是把 XML 當作一種腳本語言，它想切割的不是資料與程式，而是把程式本身切割開來，要 XML 也能下命令，分擔程式語言的工作。

所以 XML 不能做為腳本語言囉？當然也不是，像 Java EE 就很需要 XML 做為領域語言，以嚴格的語法，規範技術人員在擴展程式（或者說為系統做補丁）時只能做哪些事、不能做哪些事，直接用 Java 寫的話是無法嚴格規範的。但那是企業伺服器系統一年連續 8760 小時都不想中斷在用的，是不得不的選擇。我們平常寫程式，根本沒有不能關閉程式的需求，不需要硬加入這種擴展功能的技術。

為何選擇 Python？

Python 已然成為全民程式設計的語言，在這各國打算將程式設計納入國民教育課程的時代，Python 的流行程度，更勝過去熱門過的程式語言，有種只要辦 Python 教育課程，大家都能變得很會寫程式一樣。

確實，我認為 Python 是一門初學者在功力不夠時，就敢於開發程式的語言！他就像連鎖加盟店一樣，把所有營運流程都幫你打理好，只要加盟就教你怎樣經營一家店，降低經營失敗的風險。

但做為汎用語言的 Python，依然有它比不上其它專用語言的先天缺陷！它必須兼顧啟動的速度和運作的效率，不能像伺服端專用語言犧牲啟動的速度去換運作的效率。如果 CPython 變成每次啟動程式要等 3 秒，換來運作情況優化三倍的直譯器，那我不會想用它，因為這表示不用 0.1 秒就可以執行完的程式，卻得先花 3 秒等它優化環境再執行，這樣反而慢。所以汎用語言的 Python 是先顧好啟動速度，再兼顧運作效率。即便如此了，CPython 還是在努力解決啟動速度的問題，可見 Python 雖然不是最慢，但確實不夠快～

為何選擇 Python？它易學的特性，很適合做為程式設計新手的第一個語言。標準程式庫豐富完整、簡單好用的特性，也很適合做為程式設計老手的主力語言。然而，你會需要第二個語言，也會需要結夥其他語言。事實上，Python 的精神是做為膠水語言（glue language），用來和其他程式語言接合在一起的語言，而不是誰與爭鋒的倚天劍。

函數式程式設計（Functional programming）

函數式程式設計要先確定資料，然後再控制與處理。由於以資料為主體進行操作，所以平行化處理時很有效率。但資料失據時，函數式風格就派不上用場，因此函數式程式設計並不是萬靈丹。

傳統結構化程式設計，則是做好控制和處理，而資料其實可有可無。因此比較好貫徹的風格，其實還是以流程控制為主體的傳統結構化程式設計。函數式風格只是在派得上用場時，令人感到方便、好用罷了，但想一體適用、保持一致，並沒那麼容易。為了貫徹函數式風格，必須導入一些數學原理突破瓶頸，程式碼會變艱澀，沒有專業底子的人看不懂，這就本末倒置了[3]。

雖然現在函數式程式設計被大家掛在嘴上，好像它很了不起、很卓越，但我不認為這是比傳統結構化程式設計更好的解決問題方式。

函數式程式設計有它簡化資料處理的長處，我會在能用的地方盡量用它。但主體架構上，我還是偏重傳統結構化程式設計的解決問題方式，以結構化程式設計為主，函數式程式設計為輔，相得益彰。

物件導向程式設計（Object oriented programming）

Python 有很好的物件導向機制，但要自己撰寫類別時，重複的 self 參數、一堆 __ 雙底線的名稱，又醜又蠢，看了就煩！

我很樂於操作 Python 內建的物件，也很喜歡套用第三方的物件，但我完全不想把自己的程式寫成類別來用。

我寫 Python 的原則，就是不要寫類別，避免物件導向系統分析與設計的思維，盡量讓原始碼看起來很傳統結構化。特別是語法簡約、講究編排的 Python，能寫出乾淨、漂亮的結構化程式設計，讓我更傾向這樣寫程式。

我只有在需要自訂運算子的功能，才會寫類別去覆寫運算子。

輕量、嵌入、汎用的程式語言：Lua

當你精通 Python，開始想下一步該學什麼，好提升自己的程式設計專業，我的建議是 Lua 程式語言。

Lua 在語法上沒什麼獨特之處，就很普通的腳本語言而已，所以你不用擔心從頭學起，也不會有什麼適應期，因為語法特性比 Python 還單純。

Lua 獨到之處，在於可嵌入到其它程式使用！它很像可以編譯出 C 原生程式的腳本語言，不但執行效率高，還很容易嵌入到各裝置系統運行。不但單獨使用 Lua 語言的話，可做為進階操控電腦的腳本語言，與其他語言搭配使用的話，可做為優化程式執行效率的解決方案，解決其他程式語言在效能上的瓶頸。需要為應用軟體內建一個腳本語言供使用者進階控制時，Lua 更是不二人選。

所以 Python 學完想再練就更多武功成為絕世高手，我認為 Lua 是非常值得考慮學看看的第二門程式語言。

Python 還不能算是大眾化的程式語言

大眾化程式語言

Python 確實帶起一波程式設計的國民運動！他讓每個人都能學習當個程式設計師，有程式設計師為電腦開發功能的專長。

但 Python 能做為大眾化的程式語言嗎？不，還不行，Python 的簡單明確原則，是站在「成為一名好的程式設計師」所設計的，因此依然屬於程式設計這個行業在用的程式語言。你學會了，就是在進行程式設計師的工作。

真正大眾化的程式語言，是各行各業都能寫程式讓電腦工作的語言，非關程式設計師。醫生、律師、建築師、會計、老闆…都能寫程式，設計出自己想要的軟體，完全不需要程式設計師的專業，那樣的語言才叫大眾化！大眾化程式語言應該像鐵鎚或起子那樣的工具，每人都能拿來使用，用完收回去，而不是一份敲鐵鎚和轉起子的工作。

不是沒有這樣的大眾化程式語言，Shell Script 或 AutoHotkey 都很接近這樣的語言，但還不夠易懂易用，因此我們還要繼續努力，在有了 Python 這樣易懂易用的程式設計師語言後，進一步出現各行各業都能寫程式的大眾化語言！

Python 4？

既然 Python 3 都走到這地步了，何不期望 Python 4 朝大眾化程式語言演進？

我認為不可能，Python 的語法和規格，都是站在「程式設計師應該怎樣寫，才是好程式」的角度設計的，因此 Python 有很高的可讀性高，廣受好評。

而大眾化程式語言基本上是背道而馳的：「無關程式設計師，各行各業都能寫作。別把程式設計師那套規範硬塞給我，我不是想當程式設計師才寫程式的。」別輕忽一個事實，Python 可讀性再高，其實各行各業還是看不懂 Python 程式碼！所以別把 Python 那套可讀性硬塞給各行各業遵守。

大眾化程式語言，是一項需要創新與突破的發明！不是把 Python 重新設計就能做到的。

Python 就該是程式設計師用的語言

換個角度想，Python 也不能朝大眾化程式語言發展。

Python 必須是可讀性高的程式設計師語言，否則我們恐怕不想再用這門語言，因為大眾化程式語言的開發效率沒有比較好，只是降低門檻而已。

我只是提出 Python 依然不是大眾化程式語言。Python 雖然簡單易用，但它仍然是與程式設計師這行專業息息相關的語言，你得越來越像個程式設計師，才能發揮 Python 的能力，你無法繼續當個門外漢，卻期望能好好使用它。

我們有了 Python 這樣的程式設計師語言，但我們還沒有真正大眾化的程式語言，你不期望能有這樣的語言，把缺填起來嗎？大眾化程式語言負責寫出模塊，程式設計師語言負責寫出能跑這些模塊的系統，這種配合方式，對產業帶來的效益，應該會有將近一個級數的提升吧？

我的 Python 程式設計能力

先熟練 Java Application 和 JavaScript RIA 才開始學 Python 的我，學得很不像樣，因為習慣程式語言就是拿來開發應用程式的我，不太能適應、理解汎用程式語言這門領域。這大概是我學 BASIC、Pascal、C、C++、Java、JavaScript、JSP、Ruby、Quick Macro、AutoHotkey 以來，學得最糟糕的一次…屏除 C#、VB.NET、PHP、Go 這只學不用所以表現更差的語言[4]。

確實我能用 Python 做比 C 和 Pascal 更多的事，但我完全不覺得我很厲害，而是這語言厲害，且怎麼看都覺得每個人 Python 比我厲害。

這種感覺很像「它傻瓜，你聰明」的立可拍相機，既然專業成分降低，每個人能都拍出好相片（只要構圖別太差），用的時候就不會覺得自己高人一等。用門檻較高的單眼相機，懂得掌控快門與光圈捕捉營造的氣氛，拍相片就有種專業人士的感覺，為此沾沾自喜。Python 是每個人都能寫出好程式解決問題的語言，不像 C 或 Java 一個要懂電腦底層、一個要懂軟體工程。所以越覺得 Python 學得不怎樣，越能彰顯 Python 的價值。

無論是不是藉口，總之，我的 Python 程式設計功力並不怎樣，任何一個人都學得比我還好～
