準備

易寫易用的 Ruby 語言

用過多種程式語言的資訊工程研究生松本行弘認為，電腦應該照人的想法運作程式，而不是人配合電腦的運作苦想程式。為了讓撰寫程式是件令人愉快的事，松本行弘結合 Smalltalk 的物件導向、Perl 的靈活語法、Eiffel 的編程模式，開發了讓設計師更容易撰寫程式的 Ruby 語言。

Ruby 改善的重點並不在單字般的程式指令如 if 或 while，而是在句型般的表達式如 regular expression、slice sequences、expression substitution，因此你可用豐富多樣的句法來表達程式，使得設計程式就像寫作般抒發感想一樣，輕鬆、愉快。

不只如此，直接根據資料本身來進行程式設計的思維，而不是只能下指令控制和處理資料，讓想入門程式設計的人，即使不熟艱澀的邏輯思維，卻能因迅速掌握 Ruby 的思維，立馬著手設計程式。

Dynamic interpreter

Ruby 是動態型別的直譯式語言，因此不像靜態型別的編譯式語言，用編譯器嚴密檢查程式設計的原始碼哪裡有錯，而是寬鬆條件，讓直譯器想辦法理解錯誤的原始碼想做什麼。

所以寫 Ruby 程式可以把心力專注在你真正想要實現的功能上，不用把時間花在檢驗資料上。

由於 Ruby 著重的是寫程式，而不是跑程式，因此它的直譯器在設計上加入大量的自動化處理工作。這導致犧牲了執行效率，卻成為最聰明的直譯器。

若看重的是跑程式而不是寫程式，可以改用語法簡約，所以直譯器自動化處理的任務不多，執行效率較好一些的 Python。甚至非常著重執行效率的話，用 PHP 也不錯，這語言除了寫伺服端網頁程式，也能當一般直譯器寫應用程式來跑。沒有誰優誰劣，端看你的需求選擇出來，就是好語言。

Ruby on Rails

2005 年之前，Ruby 程式語言雖然有名，但真正用的人不多。迅速走紅的原因，是網站應用框架 Ruby on Rails，在 2006 年用它來開發網站，效率是第一的，寫沒幾行程式就能搞定 Java 落落長的程式工作，大家紛紛跳槽過去。

然而奇怪的事發生了！開發人員改用 Ruby on Rails 後，開發網站功能的效率雖然提升數十倍，但從未架過網站的新手用 Ruby on Rails，入門難度卻比 PHP、ASP.NET、JSP 還要高！

其實，Ruby on Rails 雖然把網站開發人員平常耗時費力的工作簡單化，但那些工作對新手來說還用不到。也就是說，Ruby on Rails 雖然把高難度的工作給簡單化，但這些簡單化的工作丟給新手，反而是製造麻煩了～

因此不建議想架站寫動態網頁的新手拿 Ruby on Rails 入門，還是乖乖從 PHP 學起比較平順。我們也上了一課，好用的東西不見得好學，雖然戰鬥機飛得又快又遠，但沒接受過訓練的人，還是乖乖騎著機車在附近兜風就好。

Perl v. Ruby

對我來說，Ruby 就是純物件導向且更好用的 Perl 5，光看 Ruby 連表示鍵值對資料都用 => 符號，而且都叫 hash，就覺得抄得過火了。如果遺憾 Perl 5 不夠物件導向，純物件導向的 Perl 6 又太與眾不同，那 Ruby 就是很好的選擇！

我不喜歡 Ruby 的地方，就是它太像 Perl！而抄過來的東西，在程序導向的 Perl 其實是優雅的設計，但在物件導向的 Ruby 來說，有些卻是多餘的。如果沒有那些多餘的東西，Ruby 寫起來會更流暢、順手，尤其門外漢寫 Ruby 時，會更少看到直譯器發出寫法錯誤的訊息。

因為看松本行弘當初塞這麼多 Perl 的東西進來，卻沒有經過縝密的思考為什麼，所以我不認為他是很懂程式語言的行家：「Ruby 只是他把自己要用到的語言打包在一起來用，不是什麼創新的發明，只是更方便好用的改良品。」因此網路訪談或出版書籍暢談松本行弘對程式設計的理念時，我一點興趣也沒有，覺得他被抬舉了，跟真正的大師們比還差很遠。他沒有為程式設計奠定新的學術理念，只是在既有的學術基礎上多了一種實踐，而且還亂七八糟的破壞了這些學術理念。

但後來還是看了網路訪談和出版書籍後，才發現松本行弘比這些人更貼近我們，因為他也不以大師自居，他總是和我們在同樣位置看待程式設計和語言。所以為什麼 Ruby 這麼好用？不為什麼，因為這是你我的好夥伴松本行弘設計出來的語言，而不是大師們堅持己見發明出來的語言。

Python v. Ruby

「Python 容易學習，Ruby 容易上手。」Python 的語法走簡約路線，所以不難學習就能使用。而 Ruby 的語法走實用路線[2]，每項功能的程式寫過一次，就自然記住而上手，可能不用學習就用得很順。

另外，用 Ruby 寫程式碼的速度很快，基本上把資料寫出來，再用物件導向的特性操作資料，構思程式時很流暢。Python 雖然也有資料即物件的特性，但本質上跟其它程式語言一樣，站在流程控制的角度處理資料，所以在構思程式時經常要停下來預測資料，緩慢許多。

還有，語法簡約並不表示寫出來的程式碼比較少，反而就是因為可用的技倆不多，要敲的字數反而較多。所謂 Python 語法簡約，只是要學的東西簡約，但程式碼敲出來看，往往是 Ruby 比較簡約，因為 Python 要搭配多種語法與結構才寫得出來。

「看起來 Ruby 比較好！」沒錯，對程式設計師來說，Ruby 比 Python 快狠準！確實比較好用！但對新手來說，卻不是那麼回事～

「Python 是大家的程式語言，Ruby 是程式設計師的語言。」Python 3 是汰除 Python 2 的陋習，而不向下相容的語言，好讓大家更能看懂程式碼在做什麼。照自己想法寫程式不難，但要看懂別人寫的程式是什麼想法，對新手來說可就難了。Ruby 的技術性寫法較多，語法也較凌亂，雖然也不難讀懂別人在寫什麼，但總是費時。不像學會寫 Python，就看得懂任何人寫的 Python 在做什麼，因為就像自己寫的一樣，而且靠縮排整理區塊，一行一行流暢地讀，沒有一行是沒有意義、讓人分心的 } 或 end 結尾，相當舒暢。

建議入門者先學 Python，它有更完備的程式庫可用，能做到的事比 Ruby 更多而且方便。等上手了，想追求更快的程式寫作效率，或別人寫的程式庫滿足不了你，大多程式庫都要自己寫[3]，再換 Ruby。由於兩個語言基本上很像，不用宣告變數、都用 def 設計函式，屆時要學很快：「每項功能的程式寫過一次，就自然記住而上手了，不用學習就用得很順。」

《閒聊誰比較受歡迎》

現在（2016）來看，Python 顯然是比 Ruby 受歡迎的程式語言，但十多年前可不是這麼回事～

剛進入二十一世紀，Ruby 雖然稱不上熱門，但因為很像 Perl，卻有物件導向的架構，在 CGI 捧紅 Perl 的那個年代，Ruby 還是有點名氣的。這時的 Python 還只是冷門語言，甚至當時我根本沒聽過 Python，但已經知道 Ruby 的存在。

2006 年，Python 已有名氣，但僅限於程式設計師之間的口碑流傳，真正用它寫程式的初學者不多，大多還是從 Java 或 C++ 開始學程式設計。Ruby 則因為 Ruby on Rails 的關係，CGI、PHP、Java 後端通通都閃邊站，自成第四大派系，一躍成為熱門語言，受歡迎程度更勝 Python。

相比之下，未曾爆紅的 Python，繼續只求好不求快的務實精神，持續補充程式庫的應用領域，在各領域獲得程式設計師的歡心。這時已有跡象：Ruby 好像只有 Rails 框架，而 Python 靠內建程式庫能做到的事更多。

隨著 Python 之父 Guido van Rossum 加入 Google，Python 成為這家技術領先公司內部的標準語言，許多框架都是 Python 的。別說科學計算、大數據雲、深度學習讓 Python 一紅再紅，連 Django 都循著 Python 只求好不求快的精神，以沉穩的步伐一步步改進，後來反而比越改越爛、變得不知所云的 Rails 簡單實用了！現在建議用 Django 開發網站的人，要比 Rials 還多～

這段歷史軌跡，可以看出 Python 真正的魅力不是語法本身，而是語言精神。不靠華麗的招式取勝，而靠雄厚的內功修為，吸引大家奠定深厚的程式設計功力。

但比較受歡迎的，就真的比較好嗎？不見得，俐落的 Ruby 有很多優點，不是死板的 Python 可以媲美的。如果你很清楚 Ruby 比 Python 適合你的程式寫作風格，那管他 Python 是不是更受歡迎，Ruby 對你來說才是最好的選擇！反之亦然。

Java v. Ruby

Java 必須先設計類別，製造物件，才能處理資料。

而 Ruby 的資料直接就是物件，即使不自己設計物件，也能處理得很好，甚至從頭到尾都不用自己設計類別製造物件，就能妥善完成理想的物件導向程式設計。真要設計自己的類別製造物件時，也能做得很漂亮。

不用設計類別製造物件，就能享受物件導向的美好，表示物件導向做得更妥善，誰優誰劣已經很明顯了。

但 Ruby 是動態型別語言，不適合開發大型專案，所以需要靜態型別語言來整合團隊之間的編程任務時，還是得選 Java。除非你的 Ruby 將根據某套事件驅動程式設計的框架進行開發，否則不要把 Ruby 用在百人團隊的大型專案，下場會很慘。

... v. Ruby

沒有最完美的語言，只有最適合你的語言。

只要符合你的個性、滿足你的需求，就是完美的語言。

所以你自己覺得使用的語言很完美就好，別人有別人完美的語言，沒必要去羨慕或鄙視別人使用的語言。

你喜好變了、或有更多需求無法滿足了，就去追尋新的完美語言！往前看、向前走，不用猶豫。下一個語言是否更完美不知道，但你得先向前走出完美的自己，才能往前看到完美語言的出現。

Ruby v. Ruby

Ruby 最大的對手，就是 Ruby 自己。

以 end 結束區塊，在結構層次不多時，其實很直覺也很整齊，並不難看。但四層、五層以上，整堆 end 看起來就很醜了。

然而，無論哪種語言，語法結構寫超過三層，都不會好看。在傳統結構化程式語言，會把一層層結構寫成函式，做為減少層次的手段。在語法蜜糖多的 Ruby 更是如此，可以減少層次的寫法有很多！當你看到整堆 end 很難看，其實就是警訊，告訴你程式碼要重構一下了！

連續出現五行 end 很醜不是 Ruby 的問題，而是連續寫出五行以上 end 程式碼的問題。不過很多框架，就是得連續寫出好幾行的 end，所以不見得是你的問題，而是設計上的問題～
