程式設計書評

Java 入門書籍

這是 2009 年寫的，介紹當時逛書局常見的幾本 Java 書籍。現在這些書大多再版了，所以要買的話，請自己確認有沒有新版，而且不保證新版內容如本篇所描述的一樣。

《最新 Java 2 程式設計實例入門與活用》+《Java 的真相》

如果你是電腦門外漢想學程式設計，高橋麻奈的《最新 Java 2 程式設計實例入門與活用》是不錯選擇！這本書在教導觀念時，描述得很細膩，那種體貼得無微不至的感覺，就像一位大姊姊在教你國中功課一樣。

看完這本書，再延伸閱讀前橋和彌的《Java 的真相》，對整個 Java 的想法就會超乎想像完整了！你將站在電算機原理的角度來應用程式語言，這對設計程式有很大的幫助，能讓你釐清所寫的每段程式是基於何種原理來運作。

《Java SE 6 技術手冊》

一般 Java 入門書籍，只會說明 Java 有「封裝、繼承、多型」或「多載」「泛型」等等功能可用，而林信良先生這本《Java SE 6 技術手冊》，會同時告訴你正確使用這些物件導向功能的「觀念」，讓你知道何時該用、何時不該濫用。

所以《Java SE 6 技術手冊》不只適合做為你的第一本 Java 入門書籍，也適合做為你的第二本 Java 入門書籍！

怎麼說？例如你已經買了一本 Java 入門書籍，也順利學會 Java 的各式語法功能，但實際寫程式時，一把這些功能湊在一起，卻開始遲疑：「我是知道封裝、繼承、多型怎麼寫啦，但什麼時候應該用繼承來寫？什麼情況可以寫成多型？」那就再買這本《Java SE 6 技術手冊》做為你的第二本 Java 入門書籍吧 :)

《JAVA2 程式設計從零開始》

如果喜歡 Step by step 這類簡單明瞭的圖解教學內容，何嘉益、黃世陽、李篤易、張世杰、黃鳳梅合著的《JAVA 2 程式設計從零開始》是相當棒的一本！

適合初階以上、專家未滿的玩家級電腦使用者閱讀，先求速學速用，再求實戰中累積經驗。

缺點是印刷粗糙、頁面排版的設計過於零散[1]，乍看之下實在不大像「上奇資訊」出版的電腦圖書[2]。建議實際跑一趟書店翻過後再考慮，不要直接網購來看。

《Java 程式設計藝術精要第六版》

如果你是已經懂電腦的人，想學習用程式設計來開發軟體，那 Deitel 兩兄弟合著的《Java 程式設計藝術精要第六版》很適合你。

這本書的架構，就像一本大專院校的教科書，為你安排一系列紮實的課程，相當適合自學。

《Java 編程語言 4th Edition》

如果你已經是程式設計師，想跨行學 Java 程式語言，那由 Ken Arnold、James Gosling、David Holmes 合著的《Java 編程語言 4th Edition》是最棒的選擇！

《Java 編程語言 4th Edition》並不是針對程式設計新手所編寫的書籍，而是寫給想學習 Java 的程式設計師。雖然就 Java 的部份而言，這是我所見過描述得最全面的書籍，但因為不少程式設計基本觀念不會多加解釋，所以不適合程式設計入門者閱讀。

《Thinking in Java 4/e 中文版》

如果你已經在用 Java 程式語言，Bruce Eckel 著作的《Thinking in Java 4/e 中文版》就像一本「散文集」一樣，適合閒暇之餘翻來閱讀，藉以充實想法的書籍。

這本書有價值的地方在於，它其實不適合做為 Java 入門書籍，但也不是進階的 Java 書籍，而是強化你 Java 基礎的書籍：「市面上只這本書能讓你獲得這樣的效益。」

事實上，入門以後往往發現很多小地方就算打好基礎也讓人感到茫然，通常是應用時對細節上的想法有所疑惑。這時候會陷入兩難：入門書籍沒幫助，進階書籍頭痛醫腳、搔不到癢處。這時 Bruce Eckel 著的《Thinking in Java》就是最好的「課外讀物」。

《Java 2 教學手冊 JDK5 版》

雖然衝著 Ivor Horton 的名號，我連翻都沒翻，就認定他著作的《Java 2 教學手冊 JDK5 版》肯定是經典名著，馬上敗了 850 元扛回這本厚重的書，但寫得並不好[3]。

以 Java 入門書籍來看，這本書內容不夠親切、也不夠詳實。而以綜合型的參考書來講，就算寫得再厚也不可能綜合得很全面，使得這本書不會是多出色的 Java 書籍。

不過還是適合做為綜合型的 Java 參考書，喜歡一本書能包山包海的話，這本書還是值回票價。

[1] 留白手法太糟糕，沒有將一致性的內容，好好整理成一塊，有濫用空白增加頁數，藉以維持中價位水準的嫌疑。也就是說，好好排版的話，這本書可以減少五分之一的頁數；在台灣這「以頁數計價」而不是「編輯成本計價」的地方，這本書售價應該可以更低一些。
[2] 上奇資訊在台灣近十年來，積極挖掘台灣本土資訊人才，已經出版好幾本值得台灣本土驕傲的優質著作。近幾年也積極改版中國的電腦書籍，有效提升中文電腦圖書的出版數量。在這 PC 沒落被智慧型裝置取而代之，所以電腦技術逐漸式微的時代，上奇資訊對台灣有志學習電腦技術的人，帶來很大的助益。
[3] 跟《C++ 教學範本》落差太大。《C++ 教學範本》是我這輩子體驗過最棒的程式設計入門書籍，甚至以程式語言之觀念的養成與建立來講，它遠遠超越上述任何書籍。因為這本書用盡篇幅來詳述每個語法的細節，因此你不只是學完一個 if 指令或者 for 指令的用法，而是經歷完整個 if 和 for 的所有應用情況。重點在於，篇幅雖然很長，但筆調溫柔、細膩，看完一點也不覺得累，只覺得精彩！

《無瑕的程式碼》

自從博碩推出「名家名著」系列，我開始買這些「非關技術、而是想法」的書來讀。枯燥的電腦生活獲得滋潤，因為就像在讀文學家的著作一樣，藉由文字堆砌的意境世界中，窺探大師們不凡的內在。

我還記得買第一本的《無瑕的程式碼》來看時，過著兩星期「讀聖賢書」的美好生活～

過去都是技術導向的心態研究電腦書籍，現在卻是心靈成長的感觸在閱讀電腦書籍，覺得這是很棒體驗的我，還四處向懂程式設計的朋友推薦買這本書來看！

然而，經驗上的傳遞我會覺得受益良多，技術上的取捨我不見得照單全收。例如，作者有許多類似這樣的建議：「整合開發環境已經ＯＯ，所以原始碼就不要再ＸＸ。」而我覺得把自己綁死在編輯器提供的功能是很蠢的做法，因為不保證哪天我是在跑不動這種笨重軟體的電腦寫程式、或者暫時沒有一套整合開發環境可用的機器裝置寫程式、甚至是在按鍵精靈這樣的工具底下寫程式。

確實這是一本能讓心靈成長的好書，不過未必是能讓技術進步的教科書，照單全收反而可能退步也說不定～

我不是批評這本書，我愛死這本書了，所以後來還買了《無瑕的程式碼番外篇》《Kent Beck 的實作模式》《設計模式的解析與活用》來讓想法上有更多的昇華，連 O'REILLY 的《易讀程式之美學》也買來看。只是就真的當「聖賢書」來豐富心靈，而不是「公文」來照表操課。

《我的程式碼會說話》

基本上，大師的想法歸想法，可以學到實際能用的東西一半一半而已。因為如何實踐端看個人與場合，有可能整本書介紹了十八招，看完實際能應用的卻不到五招[1]～

所以有興趣學這類「程式寫作工藝」的話，我建議《我的程式碼會說話》，先快而有效地能夠付諸實踐那些大師的理論，然後才看《無瑕的程式碼》或其他名家名著，將你已經會用的實務經驗延伸到更多境地～

《我的程式碼會說話》這本書因為薄又便宜，所以給人一種「拿別人東西東湊西湊出來、沒什麼料不值得閱讀」的感覺。但我要告訴你，他顧慮的層面比大師還廣，而薄的原因不是內容偷工減料，而是不把一句話故意寫成一整頁，所以我們一看就懂，不用看完整段故事後還要釐清情節。加上每句理論完，立刻舉簡單的程式寫法給你看，這是「名家名著」所沒有的，其實是不可多得的好書。

如果你跟我一樣，程度不夠將大師的想法付諸實踐，所以看了「名家名著」後只能拿來說嘴，那《我的程式碼會說話》能幫你實際動手做出來～

[1] 這沒什麼好訝異的，例如大師介紹他手邊的有哪些工具可以做哪些事，但一來你可能沒有、或者不需要那些工具，二來他真的只是「介紹」說可以做哪些事，卻沒教你怎麼操作：「你要自己摸索。」「大師只是告訴你該摸索哪些東西～」
[*] 我唯一嫌的，是裡面用圖文介紹如何操作軟體時，出現 OS X 視窗的介面。極度反蘋果的我看了就厭惡到極點，好幾次都有股衝動想把書給撕了 XDDD　後來我把出現 OS X 畫面的那個章節用釘書機釘住封起來，眼不見為淨。兩年後還是覺得礙眼，所以把第五章撕下來了。

《Kent Beck 的實作模式》

這本書在「前言」表示，將介紹如何表達易讀的程式碼，但內容卻跟《無瑕的程式碼》不一樣，並沒有在介紹怎樣的程式碼寫法最清晰又簡潔，而是在介紹程式碼在寫法上有哪幾種模式可循。這能讓我們更清楚自己寫物件導向程式時正在做什麼，或者說知道該透過物件導向來表達些什麼。

比如說，這本書告訴你「類別」有「值物件」「匿名內部類別」「可插拔選擇器」「庫類別」…等方式被使用，也同樣告訴你 interface 和 abstract class 有哪幾種方式被使用，在「變數」和「函式」的章節也同樣告訴你有哪幾種方式可以使用～

很多都是本來要花三年、八年的時間，才能領會到的物件導向心得，這本書都整理出來告訴你！也因此，大多項目都要很專注去讀才會懂，當散文瀏覽不容易學到東西，必須思索與品味文義裡闡述的事。甚至有兩三則因為程度不到那個境界，所以像鴨子聽雷一樣不曉得作者在講甚麼。但也不用擔心看不懂、學不完，這本書很薄，而且重點在五、六、七、八章而已，純粹是必須專注去讀而已，但其實很容易學得完、學得透[1]。

這本書應該獨立於《無瑕的程式碼》《易讀程式之美學》《我的程式碼會說話》，而且比這些書更能提升你的功力！如果你花了三年、八年的時間，還不清楚物件導向程式設計平常在做些什麼，這本書會幫你整理得清清楚楚。然後你會讚嘆 Kent Beck 真的是大師，如果沒有他出書立著，我們不可能把這些事分得這麼仔細，繼續在物件導向程式設計世界裡摸索那堆模糊的東西。

[1] 是很難讀，但很容易讀完。

《開發設計模式的秘辛：設計模式第 1¾ 版》

雖然我不主張非得套用 23 種設計模式到程式開發案例不可[1]，但我時常翻開《物件導向設計模式》，回味一下這本書四位作者萃取的智慧與經驗，以及葉秉哲為這本書翻譯時的用心之深[2]。三年前，博碩名家名著系列推出《設計模式的解析與活用》時，我也第一時間就買來讀，所以我對物件導向設計模式很有興趣，也自認為看得懂物件導向設計模式。

沒想到，我看不懂《開發設計模式的秘辛：設計模式第 1¾ 版》John Vlissides 在寫什麼，或者說我無法理解為什麼作者要基於那些想法來使用設計模式。

但我卻看得懂 Erich Gamma 和 Ralph Johnson 反對 Multicast 時的想法，而且讚嘆不已、回味再三：「這才是物件導向設計模式的精髓啊！」

所以這本書的體驗很特殊，我沒能從作者筆下學到什麼，卻從反對者意見學到更多更深的事情。

[1] 因為有時候設計模式是自然而然形成的：「設計時很自然就用到某某模式。」所以一個模式都沒用到不見得是壞設計，明明不需要卻故意套用某某模式，反而會變成壞設計。
[2] 《物件導向設計模式》中譯本品質之高，也是這本書讓我愛不釋手的原因。

《Programming Pearls, 2nd Edition 中文新修版》

看了這本書後，我的感想是：「我買了一本看不懂的書。」我數學本來就不好，所以這種很像數學家在推導問題過程，而不是開發觀念和語言技巧的書，讀起來可吃力了！

雖然無法完全看懂，但還是掌握到幾項有助於程式設計的原則。這些原則並不是新的東西，而是本來就在那兒，我卻習以為常，不曾認真思考。像是我只著重處理的部分，卻不曾在輸入和輸出上用心著墨。而好的輸入方式和輸出格式，其實是可以影響處理手法的！因為那也是問題描述的一環。總之，我還是循著書的架構，以自己的經驗去理解體會每個要點與環節。

很少程式設計師數學底子像我這麼差的，且大半寫程式的人數學底子都不錯，因此大多數人應該都能看懂這本書才對。所以看不懂的我，其實無法評論這本書寫得好不好。我只能說不適合我罷了！我比較適合讀在語法上著墨的書，而我以為這就是這樣一本書，才會買下它。

但我卻不因為自己看不懂而失望，反而愛不釋手地盡自己有限能力去讀這本書，試著汲取作者想要傳授的事情。看不懂還能看下去，努力想多學點什麼也好，我想我只能這樣傳達我對這本書的好評了！

[*] 這本書扣掉附錄，其實只有 210 頁，也就是 105 張的紙。但我花了一個多月時間，才總算把書看完，大多有看沒有懂，只是抓住方向而已。這本書讓我學到一件事：我還很嫩 XDDD

錢達智《Delphi 學習筆記 Win32 基礎篇》

念專科學校時，總覺得 Visual Basic 6 編譯出來的程式不方便散佈給他人，想改學執行檔能直接在電腦跑的 C++ 語言。

但第一次學程式語言就是 Visual Basic 的我，稱不上程式設計師，希望繼續視覺化開發環境學會的設計手法，因此媲美 Visual Biasic 的 Delphi 與 C++Builder 成為我的選項。

接著，我在圖書館借了《Delphi 學習筆記 Win32 基礎篇》這本書，內容詳實易懂，讓我有信心能學會 Delphi，於是決定以 Delphi 做為我第二的程式語言。應該說是第二個開發工具才對，因為當時我只是熟於操作 RAD 軟體，而且以為這樣就夠了，並未更專精於程式語言本身。

為了好好學習 Delphi，我希望手上也有一本《Delphi 學習筆記 Win32 基礎篇》隨時查閱與練習。但跑遍台中各大書店，就是買不到這本書。畢竟已經過了快兩年時間，在出版速度極快的推陳出新時代，書通常不會擺在架上超過三季，但當時我以為像這樣一本經典的書，應該過兩三年還買得到才對，真懷念當時的天真～

沒想到，作者在 2008 年父親節開放下載！請參考部落格「達智的東坡書房」的這篇文章：歡迎下載《Delphi 學習筆記 Win32 基礎篇》。

下載這本書一看，真實重現當年的排版樣式，就跟當年看的原書一樣，開始浮現過往的心情，真是感動啊～

雖然出社會後，我也買過幾本 Delphi 的書，但一直沒有這本《Delphi 學習筆記 Win32 基礎篇》內容詳盡、說明易懂，是我讀過最棒的 Delphi 的書籍！

其實，我並未使用 Delphi 多長的時間。用 Delphi Games Creator 這個套件，練習寫一下角色扮演遊戲後，就改用 C++Builder 了！沒多久又離開學校，為了接案，想說必須使用合法的開發工具才行，但手上只有盜版光碟的我，改用免費 JDK 寫程式，不知不覺跳脫了 RAD。

但 Delphi 的回憶，就像短暫的初戀一樣，在我青春歲月留下酸甜的滋味，可以好好回味一番～

最讓我吐血的中譯本

我看過會讓人吐血的中譯本，其實不是制定一堆讓人感到生疏的譯詞，結果書讀起來比英文還慢的「侯捷」譯作，而是「趙光正」的《UML 精華第三版：標準物件模型語言》。

我實在不太敢買侯捷翻譯的書，因為讀起來常常比英文還痛苦，往往有同類型書籍可以選的時候，我會買蔡明志翻譯的著作，文筆口語化通順很多。但身為讀者，絕對能感受到侯捷對於譯作的用心與投入，所以這只是自己適應與否的問題，沒有苛責別人的道理。（雖然還是很想建議，直接用英文名詞還比較看得懂，而不是發明一些讓人覺得彆扭、很難適應的譯詞。）

進入正題，Martin Fowler 所著《UML Distilled Third Edition》最大的好處，就是非常薄，頁次大約 200 左右，他用非常精簡的內容，介紹了龐大笨重的 UML 各式表示法，廣受好評。

但翻譯成中文的《UML 精華第三版：標準物件模型語言》，頁次居然變成 300 左右，也就是書本厚度多了 50 張紙。而多出來的部分，居然是趙光正先生大量寫了自己的「譯註」在裡面，篇幅至少達整本書的五分之一以上。而且他個人的譯註還不是放在每頁的最底端，而是夾在主文的各個段落之間。當你覺得他的譯註其實都是廢話，根本沒興趣看只想跳過時，就會越看越火大，於是成為我所看過最讓人吐血的中譯本！

原本既薄且短所以大受歡迎的優點，就這樣給破壞掉了～

中國電腦書籍的出版水準真高

台灣這邊也有出版的，像是朱印宏的《網站開發新路線 jQuery 核心詳解與實踐應用》、張亞飛的《掌握 HTML5 和 RIA 網站設計》、王洪亮的《我的程式碼會說話》…等等簡體中文翻譯書，讀過以後，覺得中國的電腦出版市場，已經在帶領有志者深入剖析技術底層、或者名師將實務驗傳授開來給高徒，台灣這邊還在出「語言入門教學」的書，深感兩岸水準的差距越來越明顯～

當然，台灣還是有陳建村先生的《笑談軟體工程：敏捷開發法的逆襲》《笑談軟體工程：例外處理設計的逆襲》帶領我們提升到進一步層次，或者黃嘉輝先生的《深入研究 Java Swing》和《深入研究 Java FX 2》至少為我們 Java 語言奠定更深的實務能力～

只是跟中國涉獵了更多領域題材比起來，台灣還是侷限在入門以上、進階未滿的領域內。

所以希望台灣名師們別再重複出「ＯＯ入門教學」的書了，分享你們的程式設計經驗與實務心得，造就我們更多高徒吧 :)

也期許兩岸共同為中文著作帶來諸子百家爭鳴的景況。
